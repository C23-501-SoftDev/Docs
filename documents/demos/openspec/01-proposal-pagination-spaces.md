# OpenSpec Proposal: Пагинация списков `spaces`

## Идентификатор фичи
`pagination-spaces-page-response`

## Контекст
Сейчас эндпоинты:
- `GET /api/spaces` (не-ADMIN)
- `GET /api/admin/spaces` (ADMIN)

принимают `page`/`size`, но отдают `List<SpaceResponse>`. Для не-ADMIN это фактически игнорирует пагинацию (возвращаются все доступные пространства).

## Цель
Сделать поведение пагинации предсказуемым и единым:
- возвращать метаданные пагинации через `PageResponse`
- для не-ADMIN корректно применять `page/size` к отсортированному списку

## Non-goals
- Изменение модели данных документов (documents/versions/attachments)
- Добавление новых эндпоинтов
- Оптимизация не-ADMIN paging “в один SQL” (на демонстрации достаточно корректного поведения)

## RFC 2119 (обязательные требования)
- `MUST` для обоих эндпоинтов возвращать тип `PageResponse<SpaceResponse>`
- `MUST` для `GET /api/admin/spaces` использовать пагинацию на уровне репозитория
- `MUST` для `GET /api/spaces` (не-ADMIN) применять `page/size` после сортировки по `createdAt DESC`
- `SHOULD` сортировать по `createdAt DESC`, как в текущей реализации репозитория
- `MAY` при `page/size` вне допустимых диапазонов возвращать пустой `content`, сохраняя корректный `totalElements`

## Ожидаемый пользовательский эффект
Клиент может корректно листать пространства:
- видеть `totalElements`, `totalPages`, `last`
- не получать весь список целиком при `page/size`
