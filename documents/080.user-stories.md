# User Stories и Acceptance Criteria

## Роль: Читатель (Reader)

### US-1: Поиск документа
**Как** Читатель, **я хочу** иметь возможность искать документы по названию и дате, **чтобы** быстро находить нужную информацию.
* **Acceptance Criteria:**
    * **Scenario 1: Поиск по названию**
        * **Given** пользователь находится на странице поиска
        * **When** пользователь вводит часть названия документа в строку поиска
        * **Then** система отображает список документов, названия которых содержат введенную строку
    * **Scenario 2: Фильтрация по датам**
        * **Given** пользователь выполнил поиск или находится в разделе документов
        * **When** пользователь задает диапазон дат создания или изменения
        * **Then** система фильтрует результаты, оставляя только документы, попадающие в указанный период
    * **Scenario 3: Ограничение прав доступа**
        * **Given** в системе есть документы из разных Пространств
        * **When** Читатель выполняет поиск
        * **Then** в результатах отображаются только те документы, к Пространствам которых у пользователя есть доступ
    * **Scenario 4: Пагинация результатов**
        * **Given** количество найденных документов превышает размер одной страницы
        * **When** пользователь просматривает результаты поиска
        * **Then** система разделяет список на страницы и предоставляет элементы навигации (пагинацию)

### US-2: Экспорт документа
**Как** Читатель, **я хочу** выгружать документ в PDF, **чтобы** иметь возможность изучить его офлайн или распечатать.
* **Acceptance Criteria:**
    * **Scenario 1: Доступность функции экспорта**
        * **Given** пользователь открыл страницу просмотра документа
        * **Then** на странице отображается кнопка "Экспорт"
    * **Scenario 2: Выбор формата экспорта**
        * **When** пользователь нажимает кнопку "Экспорт"
        * **Then** система предлагает выбрать один из форматов: PDF, DOCX, HTML
    * **Scenario 3: Сохранение структуры при экспорте**
        * **Given** документ содержит Wiki-разметку (заголовки, списки, таблицы)
        * **When** пользователь экспортирует документ в любой из доступных форматов
        * **Then** сформированный файл полностью сохраняет визуальную структуру и иерархию исходной разметки

## Роль: Редактор (Editor)

### US-3: Создание документа по шаблону
**Как** Редактор, **я хочу** использовать готовые шаблоны (Архитектура, Требования), **чтобы** соблюдать стандарты документирования.
* **Acceptance Criteria:**
    * **Scenario 1: Выбор роли при создании**
        * **Given** Редактор инициирует создание нового документа
        * **When** открывается форма создания
        * **Then** система требует выбрать роль (Разработчик, Аналитик, Администратор) для определения набора шаблонов
    * **Scenario 2: Фильтрация шаблонов по ролям**
        * **Given** пользователь выбрал конкретную роль
        * **When** он открывает список доступных шаблонов
        * **Then** отображаются только те шаблоны, которые привязаны к выбранной роли
    * **Scenario 3: Автогенерация номера требования**
        * **Given** выбран шаблон "Требования к системе"
        * **When** документ создается или наполняется контентом
        * **Then** система автоматически присваивает и отображает уникальный порядковый номер требования
    * **Scenario 4: Статус черновика**
        * **When** новый документ сохранен впервые
        * **Then** ему автоматически присваивается статус `Draft`, и он остается в нем до явного действия "Опубликовать"

### US-4: Управление версиями
**Как** Редактор, **я хочу** видеть историю изменений и иметь возможность отката, **чтобы** исправить ошибочные правки.
* **Acceptance Criteria:**
    * **Scenario 1: Просмотр истории версий**
        * **Given** документ редактировался несколько раз
        * **When** пользователь открывает историю изменений
        * **Then** система выводит список всех версий с указанием автора правки и точного времени сохранения
    * **Scenario 2: Сравнение версий (diff)**
        * **Given** пользователь выбрал две разные версии документа
        * **When** активируется режим сравнения
        * **Then** система визуально выделяет добавленные, измененные и удаленные фрагменты текста
    * **Scenario 3: Безопасный откат версии**
        * **Given** пользователь инициирует откат к одной из предыдущих версий
        * **When** операция подтверждена
        * **Then** система создает новую текущую версию, идентичную выбранной старой, сохраняя при этом всю предыдущую историю изменений без удалений

## Роль: Администратор (Admin)

### US-5: Управление пространствами
**Как** Администратор, **я хочу** создавать Пространства и назначать права, **чтобы** структурировать базу знаний по проектам.
* **Acceptance Criteria:**
    * **Scenario 1: CRUD операции над Пространствами**
        * **Given** пользователь имеет права Администратора
        * **When** он переходит в панель управления
        * **Then** ему доступны функции создания, редактирования параметров и удаления Пространств
    * **Scenario 2: Назначение ответственного**
        * **Given** открыта форма настроек Пространства
        * **When** Администратор выбирает пользователя из списка
        * **Then** этот пользователь назначается "Ответственным редактором" данного Пространства
    * **Scenario 3: Настройка прав доступа**
        * **Given** Администратор настраивает безопасность Пространства
        * **When** он выбирает группу пользователей и уровень доступа (Чтение или Запись)
        * **Then** права применяются ко всем документам внутри этого Пространства для участников выбранной группы
