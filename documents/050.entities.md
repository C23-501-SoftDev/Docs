# Сущности и их жизненный цикл (Entities & Lifecycle)

## Основные сущности (Aggregates)

### 1. Пользователь (User)
Субъект системы, имеющий доступ к базе знаний.

**Атрибуты:** 
- `id` — уникальный идентификатор
- `login` — логин для входа
- `password_hash` — хеш пароля (BCrypt)
- `email` — адрес электронной почты
- `role` — глобальная роль (`GUEST`, `READER`, `EDITOR`)
- `is_admin` — флаг администратора (boolean, по умолчанию `false`)
- `created_at` — дата создания
- `updated_at` — дата последнего обновления

**Связи:** 
- Создает `Document`
- Управляет `Space` (как владелец)
- Имеет `SpaceUserPermission` (личные права доступа к пространствам)
- Имеет `DocumentPermission` (дополнительные права на документы)
- Входит в `UserGroup` (через `UserGroupMember`)
- Создает `Version` (версии документов)
- Загружает `Attachment` (вложения)

**Модель ролей:**
- `GUEST` — только вход в систему. Все права на контент предоставляются через `SpaceUserPermission`, `SpaceGroupPermission` (через группы) и `DocumentPermission`.
- `READER` — глобальное право на чтение любых документов в системе (без необходимости назначения прав на уровне пространства).
- `EDITOR` — глобальное право на создание, редактирование и удаление любых документов в системе.
- `is_admin = true` — доступ к панели администратора, управление пользователями, группами, пространствами и правами доступа. Не дает автоматических прав на контент (они определяются ролью).
- Пользователь может иметь комбинацию роли и флага администратора (например, `GUEST` + `is_admin` = "чистый администратор" без прав на контент; `EDITOR` + `is_admin` = редактор с административным доступом).

**Примечание:** Пользователи не поддерживают soft-удаление. При необходимости блокировки доступа используется смена роли.

---

### 2. Пространство (Space)
Контейнер для документов с едиными настройками прав доступа.

**Атрибуты:**
- `id` — уникальный идентификатор
- `name` — название пространства
- `description` — описание
- `owner_id` — ответственный редактор (ссылка на `User`)
- `created_at` — дата создания
- `updated_at` — дата последнего обновления

**Связи:**
- Содержит множество `Document`
- Имеет множество `SpaceUserPermission` (личные права пользователей)
- Имеет множество `SpaceGroupPermission` (права групп)
- Принадлежит `User` (владельцу)

**Примечание:** Пространства не поддерживают soft-удаление. Удаление пространства запрещено, если в нём есть документы (`ON DELETE RESTRICT`).

---

### 3. Личное право доступа к пространству (SpaceUserPermission)
Назначает конкретному пользователю личные права на конкретное пространство.

**Атрибуты:**
- `id` — уникальный идентификатор
- `space_id` — ссылка на `Space`
- `user_id` — ссылка на `User`
- `permission_type` — тип права (`READ`, `WRITE`, `OWNER`)
- `granted_at` — дата выдачи права

**Связи:**
- Принадлежит `Space`
- Принадлежит `User`

**Примечание:** При удалении пользователя или пространства права доступа удаляются автоматически (`ON DELETE CASCADE`).

---

### 4. Группа пользователей (UserGroup)
Именованное объединение пользователей для массового назначения прав на пространства.

**Атрибуты:**
- `id` — уникальный идентификатор
- `name` — название группы
- `description` — описание группы
- `created_at` — дата создания
- `updated_at` — дата последнего обновления

**Связи:**
- Содержит множество `UserGroupMember` (участники группы)
- Имеет множество `SpaceGroupPermission` (права группы на пространства)

**Примечание:** Группы создаются и управляются только администратором (`is_admin = true`). Пользователь может входить в несколько групп одновременно или не входить ни в одну. При удалении группы её участники не удаляются — связи в `user_group_members` удаляются каскадно (`ON DELETE CASCADE`).

---

### 5. Участник группы (UserGroupMember)
Связь "многие-ко-многим" между пользователями и группами.

**Атрибуты:**
- `id` — уникальный идентификатор
- `group_id` — ссылка на `UserGroup`
- `user_id` — ссылка на `User`
- `added_at` — дата добавления в группу

**Связи:**
- Принадлежит `UserGroup`
- Принадлежит `User`

**Ограничения:** `UNIQUE(group_id, user_id)` — пользователь не может дважды входить в одну и ту же группу.

**Примечание:** При удалении пользователя или группы связь удаляется каскадно (`ON DELETE CASCADE`).

---

### 6. Право доступа группы к пространству (SpaceGroupPermission)
Назначает группе пользователей права на конкретное пространство.

**Атрибуты:**
- `id` — уникальный идентификатор
- `space_id` — ссылка на `Space`
- `group_id` — ссылка на `UserGroup`
- `permission_type` — тип права (`READ`, `WRITE`, `OWNER`)
- `granted_at` — дата выдачи права

**Связи:**
- Принадлежит `Space`
- Принадлежит `UserGroup`

**Ограничения:** `UNIQUE(space_id, group_id, permission_type)`.

**Примечание:** При удалении группы или пространства права удаляются автоматически (`ON DELETE CASCADE`).

---

### 7. Шаблон документа (Template)
Предустановленная структура документа для определенной роли.

**Атрибуты:**
- `id` — уникальный идентификатор
- `name` — название шаблона
- `description` — описание
- `content` — содержимое шаблона в Markdown
- `role` — роль, для которой доступен (`Admin`, `Editor`, `Reader`, `Developer`, `Analyst`)
- `is_system` — системный шаблон (нельзя изменить/удалить)
- `created_at` — дата создания
- `updated_at` — дата последнего обновления

**Связи:**
- Используется в множестве `Document`

**Примечание:** На этапе MVP шаблоны встроены в систему, администратор не может их менять (`is_system = TRUE`).

---

### 8. Документ (Document)
Центральная сущность системы, основная единица хранения информации.

**Атрибуты:**
- `id` — уникальный идентификатор
- `title` — заголовок документа
- `content` — содержимое в формате Wiki-разметки (Markdown)
- `status` — статус жизненного цикла (`Draft`, `Published`, `Deleted`). При переходе в `Deleted` документ считается мягко удалённым и скрывается из результатов поиска и списков, но физически остаётся в БД и Git-репозитории.
- `author_id` — ссылка на автора (`User`)
- `space_id` — ссылка на пространство (`Space`)
- `template_id` — ссылка на использованный шаблон (`Template`)
- `parent_document_id` — ссылка на родительский документ (для иерархии)
- `created_at` — дата создания
- `updated_at` — дата последнего изменения (включая изменение статуса)

**Связи:**
- Принадлежит `Space`
- Создан `User` (автором)
- Использует `Template` (опционально)
- Может иметь родительский `Document` (иерархия)
- Может содержать дочерние `Document`
- Имеет список `Attachment`
- Имеет историю `Version`
- Имеет список `DocumentPermission`

**Примечание:** Документы поддерживают soft-удаление через перевод в статус `Deleted`. Все связанные данные (версии, вложения) при этом сохраняются.

---

### 9. Версия (Version)
Снимок состояния документа, связанный с Git-коммитом.

**Атрибуты:**
- `id` — уникальный идентификатор
- `document_id` — ссылка на `Document`
- `git_hash` — SHA-1 хэш коммита в Git-репозитории
- `author_id` — ссылка на автора правки (`User`)
- `comment` — сообщение коммита
- `restored_from_version_id` — ссылка на версию, из которой выполнен откат
- `created_at` — дата создания версии

**Связи:**
- Принадлежит `Document`
- Создана `User` (автором)
- Может быть восстановлена из другой `Version`
- Имеет список `Attachment` (привязанных к этой версии)

**Примечание:** Версии удаляются каскадно вместе с документом при `hard-delete`. При `soft-delete` (статус `Deleted`) версии сохраняются и доступны для просмотра администратором.

---

### 10. Вложение (Attachment)
Файл, связанный с документом (изображения, PDF, архивы и т.д.).

**Атрибуты:**
- `id` — уникальный идентификатор
- `document_id` — ссылка на `Document`
- `version_id` — ссылка на `Version` (NULL = актуальное для последней версии)
- `filename` — оригинальное имя файла
- `content_type` — MIME-тип
- `size_bytes` — размер в байтах
- `storage_path` — путь к файлу в blob-хранилище
- `uploaded_at` — дата загрузки
- `uploaded_by` — ссылка на загрузившего пользователя (`User`)

**Связи:**
- Принадлежит `Document`
- Принадлежит `Version` (опционально)
- Загружено `User` (может быть NULL, если пользователь удалён)

**Примечание:** Вложения удаляются каскадно вместе с документом при `hard-delete`. При `soft-delete` документа вложения сохраняются. Привязка к конкретной версии позволяет отслеживать изменения вложений во времени.

---

### 11. Право доступа к документу (DocumentPermission)
Дополнительные права на конкретный документ (поверх прав пространства и глобальной роли).

**Атрибуты:**
- `id` — уникальный идентификатор
- `document_id` — ссылка на `Document`
- `user_id` — ссылка на `User`
- `permission_type` — тип права (`READ`, `WRITE`, `OWNER`)
- `granted_by` — ссылка на пользователя, выдавшего право
- `granted_at` — дата выдачи права

**Связи:**
- Принадлежит `Document`
- Принадлежит `User` (кому выдано)
- Принадлежит `User` (кто выдал)

**Примечание:** Позволяет автору документа открыть доступ другим пользователям или ограничить его в рамках прав, заданных на уровне пространства.

---

## Правило вычисления прав на пространство

Для каждого пользователя итоговое право на пространство определяется как **максимальное** из следующих источников:

1. **Глобальная роль:** `EDITOR` дает право `WRITE` на все пространства (включая создание/удаление). `READER` дает право `READ` на все пространства. `GUEST` не дает глобальных прав.
2. **Личные права:** Права, назначенные пользователю напрямую через `SpaceUserPermission`.
3. **Групповые права:** Права всех групп, в которых состоит пользователь, через `SpaceGroupPermission`.

**Приоритет прав:** `OWNER` > `WRITE` > `READ`.

**Пример:** Пользователь имеет глобальную роль `GUEST`, личное право `READ` на пространство, и состоит в группе с правом `WRITE` на это же пространство. Итоговое право — `WRITE`. Если пользователь исключается из группы, его право снижается до `READ` (личное). Если группа удаляется — аналогично.
