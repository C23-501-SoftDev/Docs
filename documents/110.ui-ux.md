# UI/UX Прототипирование (Требования к интерфейсу)

## Общие принципы
* **Desktop-first:** Система оптимизирована для использования на настольных ПК.
* **Чистый интерфейс:** Минимум лишних элементов, фокус на контенте документа.
* **Контекстное управление:** Кнопки "Редактировать", "Удалить" отображаются только при наличии прав (роль `EDITOR` или право `WRITE`/`OWNER` на пространство/документ). Кнопка "Создать" отображается для пользователей с ролью `EDITOR` или правом `WRITE`/`OWNER` в выбранном пространстве.

## Принципы UX-дизайна
* **Консистентность:** Одинаковые элементы интерфейса работают одинаково во всей системе.
* **Обратная связь:** Любое действие пользователя сопровождается откликом (уведомление, изменение состояния).
* **Предсказуемость:** Поведение интерфейса должно быть очевидным.
* **Минимизация когнитивной нагрузки:** Пользователь не должен запоминать лишнюю информацию.
* **Доступность:** Интерфейс должен быть удобен для пользователей с разными возможностями.

## Основные экраны

### 1. Главная страница / Список документов
* **Левая панель:** Иерархическое дерево пространств и документов (оглавление). Отображаются только пространства, к которым у пользователя есть доступ (через глобальную роль `READER`/`EDITOR`, личные права `SpaceUserPermission` или права групп `SpaceGroupPermission`).
* **Центральная часть:** Список документов с пагинацией.
* **Верхняя панель:** Поисковая строка, фильтры по дате, кнопка "Создать" (для пользователей с ролью `EDITOR` или правом `WRITE`/`OWNER` в пространстве).

**Состояния интерфейса:**
* Пустой список — сообщение "Нет документов"
* Загрузка — индикатор загрузки
* Ошибка — сообщение с возможностью повторить запрос

---

### 2. Страница просмотра документа
* **Заголовок:** Название документа, автор, дата последнего изменения, статус.
* **Тело:** Отрендеренный Wiki-текст.
* **Правая панель (опционально):** Вложения и связанные документы.
* **Действия:** Кнопки "Редактировать", "Экспорт", "История версий" — отображаются в зависимости от прав пользователя:
  * "Редактировать" — только для `EDITOR` или при наличии права `WRITE`/`OWNER` на документ/пространство
  * "Экспорт" — для всех, кто имеет доступ к чтению документа
  * "История версий" — для всех, кто имеет доступ к чтению документа

**UX-особенности:**
* Быстрый переход к редактированию
* Видимые изменения статуса документа
* Удобная навигация по содержимому

---

### 3. Редактор документа
* **Двухпанельный режим:** Слева — ввод Wiki-разметки, справа — предпросмотр в реальном времени.
* **Панель атрибутов:** Поля для ввода метаданных (в зависимости от шаблона).
* **Управление вложениями:** Drag-and-drop загрузка файлов.

**UX-особенности:**
* Автосохранение
* Подсветка синтаксиса
* Предупреждение при выходе без сохранения
* Быстрое переключение между режимами

---

### 4. История версий
* **Список:** Таблица с версиями (дата, автор, комментарий).
* **Режим сравнения:** Две колонки с подсветкой изменений (красный — удалено, зеленый — добавлено).

**UX-особенности:**
* Возможность отката к предыдущей версии (только для `EDITOR` или при наличии права `WRITE`/`OWNER`)
* Удобное сравнение изменений

---

### 5. Панель администратора
* **Доступ:** Только для пользователей с `is_admin = true`. Ссылка на панель в навигации скрыта для остальных.
* **Разделы:** Пользователи, Группы, Пространства, Конфигурации.

#### 5.1. Управление пользователями
* **Таблица пользователей:** Колонки: `ID`, `Логин`, `Email`, `Роль`, `Администратор`, `Дата создания`, `Действия`.
* **Форма создания/редактирования пользователя:**
  * Поле `Логин` (текстовое, обязательное, уникальное)
  * Поле `Email` (email, обязательное, уникальное)
  * Поле `Пароль` (пароль, обязательно при создании, опционально при редактировании)
  * Выпадающий список `Роль`: `GUEST`, `READER`, `EDITOR` (обязательное)
  * Чекбокс `Администратор` (`is_admin`) — дает доступ к панели администратора
  * **Отображение ролей:** В таблице и карточке пользователя роль отображается как `GUEST` / `READER` / `EDITOR`. Если `is_admin = true`, дополнительно отображается метка "Администратор".
* **Валидация:** Администратор не может снять флаг `is_admin` с самого себя (защита от потери доступа).
* **Блокировка пользователя:** Осуществляется сменой роли на `GUEST` (при этом `is_admin` также сбрасывается).

#### 5.2. Управление пространствами
* Создание, редактирование, удаление пространств.
* Назначение владельца пространства.
* Назначение прав доступа для пользователей и групп (`READ`, `WRITE`, `OWNER`).
* **Две вкладки при настройке прав:**
  * **Пользователи:** Выбор пользователя → назначение права.
  * **Группы:** Выбор группы → назначение права.

#### 5.3. Управление правами
* **Назначение прав на пространства:** Выбор пространства → выбор пользователя или группы → выбор типа права (`READ`, `WRITE`, `OWNER`).
* **Назначение прав на документы:** Доступно администратору и владельцу документа.

#### 5.4. Управление группами пользователей
* **Доступ:** Только для пользователей с `is_admin = true`. Отдельный пункт в боковом меню панели администратора.
* **Таблица групп:** Колонки: `ID`, `Название`, `Описание`, `Количество участников`, `Дата создания`, `Действия`.
* **Форма создания/редактирования группы:**
  * Поле `Название` (текстовое, обязательное, уникальное)
  * Поле `Описание` (текстовое, опциональное)
* **Управление составом группы (отдельная страница или модальное окно):**
  * **Список участников:** Таблица с колонками `ID`, `Логин`, `Email`, `Роль`, `Действия` (кнопка "Исключить").
  * **Добавление участника:** Выпадающий список всех пользователей системы (с поиском по логину) → кнопка "Добавить". Пользователь, уже состоящий в группе, не отображается в списке для добавления.
* **Удаление группы:** Кнопка "Удалить" с подтверждением. При удалении группы пользователи не затрагиваются — удаляются только связи и права группы на пространства.
* **Состояния интерфейса:**
  * Пустой список групп — сообщение "Нет групп пользователей"
  * Пустой состав группы — сообщение "В группе нет участников"
  * Загрузка — индикатор загрузки
  * Ошибка — сообщение с возможностью повторить запрос

## Требования к доступности (Accessibility)
* Поддержка горячих клавиш (Ctrl+S — сохранить, Ctrl+F — поиск)
* Контрастная цветовая схема
* Поддержка навигации с клавиатуры
* Читаемые размеры шрифтов
* Поддержка screen reader (ARIA-атрибуты)

---

## Состояния интерфейса (глобальные)
* **Loading:** отображение индикатора загрузки
* **Empty:** отсутствие данных (информирование пользователя)
* **Error:** сообщение об ошибке с возможностью повторного действия
* **Success:** подтверждение успешного действия

---

## Базовые UX-трансформации

### 1. Упрощение интерфейса
Удаление лишних элементов и сокращение количества действий пользователя.

**Пример:**
- Было: форма с большим количеством полей  
- Стало: только обязательные поля, остальные скрыты

---

### 2. Группировка элементов
Связанные элементы располагаются рядом.

**Пример:**
- Поля адреса объединены в один блок

---

### 3. Консистентность
Одинаковые элементы интерфейса имеют единый стиль и поведение.

---

### 4. Обратная связь
Система информирует пользователя о результате действий.

**Пример:**
- "Документ сохранён"
- Индикатор загрузки

---

### 5. Минимизация когнитивной нагрузки
Интерфейс не требует от пользователя запоминания информации.

---

### 6. Доступность
Интерфейс адаптирован для пользователей с разными возможностями.

---

### 7. Выравнивание и визуальная иерархия
Элементы интерфейса выровнены и логически структурированы для улучшения восприятия информации.

---

### 8. Интеллектуальный поиск и ввод
Минимизация усилий пользователя при вводе данных через механизмы подсказок и оптимизацию запросов.

* **Автодополнение (Autocomplete):** Предложение вариантов из базы данных в процессе набора текста.
* **Триггер ввода (Min length):** Запуск поиска только после ввода минимально достаточного количества символов (обычно 3) для сокращения нерелевантных результатов.
* **Оптимизация запросов (Debounce):** Задержка отправки запроса на сервер до момента паузы в наборе текста пользователем для снижения нагрузки на систему.
