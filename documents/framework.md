# Kiro (AWS), Spec Kit, OpenSpec и OpenCode: сравнение в контексте ИИ-агентов

Документ фиксирует четыре экосистемных направления, которые **имеют смысл только при работе с ИИ-агентами** (автономные или встроенные в IDE). Чисто «документные» процессы без агента здесь не рассматриваются.

**Уточнение по названиям.** Под «Open Spec» обычно имеют в виду **OpenSpec** ([Fission-AI/OpenSpec](https://github.com/Fission-AI/OpenSpec/), [openspec.dev](https://openspec.dev/)) — фреймворк spec-driven для ассистентов в репозитории. Это **не** стандарт **Agent Skills** (Anthropic: каталог `SKILL.md` и т.п.) — отдельная тема про расширяемость агентов, а не про SDD-артефакты в git.

---

## Рекомендуемый фреймворк (по официальной документации)

В открытой документации сравнение четырёх направлений **явно формулирует** авторы **OpenSpec** в корневом [README](https://github.com/Fission-AI/OpenSpec/blob/main/README.md) (раздел *How we compare* и *Usage Notes*).

| Альтернатива | Как позиционирует OpenSpec |
|--------------|----------------------------|
| **Spec Kit (GitHub)** | «Тщательный, но тяжёлый»: жёсткие фазовые ворота, много Markdown, настройка Python. OpenSpec — **легче** и допускает **свободные итерации** без жёсткой церемонии. |
| **Kiro (AWS)** | Мощный продукт, но **привязка к их IDE** и ограничения по моделям. OpenSpec — **с любыми уже используемыми инструментами**. |
| **Без спек-слоя** | Промпты без зафиксированных требований дают непредсказуемый результат; OpenSpec добавляет **предсказуемость** без избыточной формальности. |

**Рекомендации по моделям** (тот же README, *Usage Notes*): для планирования и реализации OpenSpec **рекомендует** сильные reasoning-модели — в документе указаны **Claude Opus 4.5** и **GPT 5.2** (актуальные имена моделей нужно сверять с провайдером на момент внедрения). Дополнительно: чистое контекстное окно перед имплементацией и «гигиена» контекста в сессии.

**Итог для практики:** при типичном сценарии **«репозиторий + любой из поддерживаемых агентов»** официальная линия OpenSpec — выбрать **OpenSpec** как слой спецификаций; **Kiro** — если осознанно уходите в экосистему AWS IDE; **Spec Kit** — если нужен именно **жёсткий** многофазный pipeline в стиле GitHub; **OpenCode** — как **открытый агент** (CLI, MCP), который можно сочетать со **Spec Kit** или **OpenSpec**.

Текущий рабочий поток в CLI OpenSpec: `openspec init` → slash-команды вида `/opsx:propose`, `/opsx:apply`, `/opsx:archive` (подробности — [docs/commands.md](https://github.com/Fission-AI/OpenSpec/blob/main/docs/commands.md) и [Getting Started](https://github.com/Fission-AI/OpenSpec/blob/main/docs/getting-started.md)).

## Сводная таблица

| Критерий | **Kiro (AWS)** | **Spec Kit (GitHub)** | **OpenSpec** | **OpenCode** |
|----------|----------------|----------------------|--------------|--------------|
| **Роль** | Продукт: IDE (форк VS Code) + встроенный агент | Набор практик и артефактов + CLI `specify`; агент снаружи | Лёгкий SDD в репозитории + CLI; агент снаружи | Открытый **агент** (CLI/TUI), много провайдеров моделей |
| **Где живёт «спека»** | В проекте + steering/hooks в модели Kiro | В репо после `specify init`, slash-команды | `openspec/specs/`, изменения в `openspec/changes/...` | По желанию; фокус на сессиях/инструментах, не обязателен SDD-дерево как у OpenSpec |
| **Типичный поток** | Спека → план → реализация → деплой в экосистеме AWS | `/constitution` → `/specify` → `/clarify` → `/plan` → `/tasks` → `/analyze` → `/implement` | Proposal → design → tasks → дельты спеков (Given/When/Then, RFC 2119) | `opencode run`, TUI, `serve`/`web`, MCP, подключаемые агенты |
| **Привязка к облаку** | Сильная (Amazon Bedrock, сервисы AWS) | Нет | Нет | Нет (сам хостинг, ключи провайдеров) |
| **Лицензия / открытость** | Проприетарный продукт AWS | Open source (MIT), [github/spec-kit](https://github.com/github/spec-kit) | Open source, npm-пакет `@fission-ai/openspec` | Open source, много LLM провайдеров |

## Kiro (AWS)

- **Что это:** спек-драйв IDE с **встроенным агентом**, рассчитанным на длительную автономную работу, координацию репозиториев и интеграцию с AWS (в т.ч. Bedrock).
- **Агенты:** основной сценарий — **сам Kiro как единый продуктовый агент** внутри среды, а не «подключите любой CLI». Для команд важны guardrails (доступ к прод-средам, ревью спеков и PR).
- **Отличие от остальных трёх:** это не «библиотека команд для Cursor», а **отдельная среда разработки** под AWS-стек.

Документация: [AWS — Kiro](https://aws.amazon.com/documentation-overview/kiro/), [kiro.dev — autonomous agent](https://kiro.dev/autonomous-agent/).

---

## Spec Kit (GitHub)

- **Что это:** **спека как управляемый конвейер** через slash-команды в уже выбранном агенте/редакторе; инициализация через CLI `specify` (`uv` и т.д.).
- **Агенты:** проект явно рассчитан на **внешних** coding-агентов. На сайте перечислены, среди прочих: **Claude Code**, **GitHub Copilot**, **Cursor**, **Gemini CLI**, **Windsurf**, **Codex CLI** (ограниченно), **OpenCode**, **Qwen Code**, **Kilo Code**, **Auggie CLI**, **Roo Code** и др. — см. [speckit.org](https://speckit.org/) и README репозитория.
- **Сильная сторона:** выверенный порядок фаз (конституция → спека → уточнение → план → задачи → анализ → имплементация).

Репозиторий: [github/spec-kit](https://github.com/github/spec-kit).

---

## OpenSpec (Fission-AI)

- **Что это:** **лёгкий SDD в git**: каталог спеков по возможностям, изменения как «дельты» требований, артефакты proposal/design/tasks для ревью до кода.
- **Агенты:** не поставляет своего рантайма; рассчитан на ассистентов вроде **Claude Code**, **Cursor**, **Codex**, **Copilot**, **OpenCode**, **Windsurf**, **Gemini CLI**, **Cline**, **Roo Code**, **Amazon Q** и длинный список на [openspec.dev](https://openspec.dev/) — работа через согласованные промпты/команды в этих инструментах.
- **Отличие от Spec Kit:** акцент на **живой библиотеке спеков** и диффах требований (удобно ревьюить намерение), а не на фиксированном наборе slash-команд GitHub.

Установка: `npm install -g @fission-ai/openspec@latest`. Репозиторий: [Fission-AI/OpenSpec](https://github.com/Fission-AI/OpenSpec/).

---

## OpenCode

- **Что это:** **open-source агент** для кода: CLI, TUI, режимы `serve`/`web`, поддержка **MCP**, множество провайдеров моделей.
- **Агенты:** **OpenCode сам является средой выполнения агента**; внутри — управление конфигурацией, сессиями, под-командами вроде `opencode agent`, интеграции (в т.ч. GitHub). Его можно использовать **вместе** с OpenSpec/Spec Kit как один из «движков», для которых эти фреймворки генерируют контекст и процесс.
- **Сильная сторона:** единый открытый CLI, приватность/самохостинг, гибкость модели.

Документация: [open-code.ai/docs](https://open-code.ai/docs/en/cli) (и зеркала вроде opencode.ai — у проекта несколько доменов документации).

## Какие агенты считать «основными» (ландшафт 2026)

Ниже — не исчерпывающий список, а **часто встречающиеся опоры экосистемы**, с которыми совместимы Spec Kit и OpenSpec, и которые конкурируют/дополняют узкоспециализированные IDE вроде Kiro:

| Категория | Примеры |
|-----------|---------|
| **Редактор + встроенный агент** | [Cursor](https://cursor.com), [Windsurf](https://windsurf.com), GitHub Copilot в VS Code |
| **CLI / терминальный агент** | [Claude Code](https://www.anthropic.com) (Anthropic), [OpenCode](https://open-code.ai), **Gemini CLI** (Google), **Codex CLI** (OpenAI), **Amazon Q** Developer |
| **Спека-first / оркестрация репозитория** | Процессы **Spec Kit** и **OpenSpec** поверх любого из агентов выше |
| **Облачный IDE + длинный автономный цикл** | **Kiro** как отдельный продукт AWS |

Индустриально чаще всего в сравнениях и интеграциях фигурируют **Claude (в т.ч. Claude Code)**, **Cursor**, **GitHub Copilot** и **OpenCode** как представители «генеральной линии»; остальные — по региону, лицензии и облаку (Google/OpenAI/AWS).

## Краткие рекомендации по выбору

- Нужна **глубокая связка с AWS и единая среда «спека + код + деплой»** — смотреть **Kiro** и политики доступа.
- Нужен **стандартизированный GitHub-стиль pipeline из slash-команд** — **Spec Kit**.
- Нужна **библиотека требований в репо и ревью дельт спеков** — **OpenSpec**.
- Нужен **открытый, мультипровайдерный агент в терминале** с MCP — **OpenCode**; при желании комбинировать со Spec Kit или OpenSpec.
