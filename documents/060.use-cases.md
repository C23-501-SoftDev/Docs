# Use Case диаграммы и описания (Use Cases)

## Диаграмма вариантов использования

```mermaid
usecaseDiagram
    actor "Гость (GUEST)" as Guest
    actor "Читатель (READER)" as Reader
    actor "Редактор (EDITOR)" as Editor
    actor "Администратор (ADMIN)" as Admin

    Guest --> (Авторизация)
    
    Reader --> (Авторизация)
    Reader --> (Поиск документа)
    Reader --> (Просмотр документа)
    Reader --> (Экспорт документа)
    Reader --> (Навигация по оглавлению)

    Editor --> (Создание документа по шаблону)
    Editor --> (Редактирование документа)
    Editor --> (Управление вложениями)
    Editor --> (Управление версиями)
    Editor --> (Soft-удаление документа при отсутствии активных дочерних)

    Admin --> (Управление пользователями)
    Admin --> (Soft-удаление пользователя)
    Admin --> (Восстановление пользователя)
    Admin --> (Управление группами пользователей)
    Admin --> (Управление пространствами)
    Admin --> (Назначение прав доступа)
    Admin --> (Настройка конфигураций сред)
    Admin --> (Восстановление документов)
    Admin --> (Hard-delete документов при отсутствии любых дочерних)

    (Создание документа по шаблону) ..> (Авторизация) : include
    (Редактирование документа) ..> (Авторизация) : include
    (Управление пользователями) ..> (Авторизация) : include
    (Управление группами пользователей) ..> (Авторизация) : include
```

**Примечания:** 
- Администратор (`ADMIN`) — это пользователь с `is_admin = true`. Его права на контент определяются ролью (`GUEST`/`READER`/`EDITOR`). На диаграмме показаны только административные функции, доступные при `is_admin = true`.
- **Примечание:** Soft-удаление документа блокируется, если у него есть дочерние документы в статусах `Draft` или `Published`. Hard-delete блокируется при наличии любых дочерних документов. Дочерний документ должен принадлежать тому же пространству, что и родительский.

## Описание ключевых процессов

### UC-1: Создание документа по шаблону
* **Действующее лицо:** Редактор (пользователь с ролью `EDITOR`) или пользователь с правом `WRITE`/`OWNER` в пространстве.
* **Предусловие:** Пользователь авторизован в системе.
* **Основной поток:**
    1. Редактор нажимает кнопку "Создать документ".
    2. Система предлагает выбрать тип шаблона (Архитектура, Требования, Инструкция и т.д.).
    3. Редактор выбирает шаблон.
    4. Система открывает форму с предзаполненными атрибутами и полем для Wiki-разметки.
    5. Редактор заполняет данные и выбирает Пространство.
    6. Редактор нажимает "Сохранить".
    7. Система создает документ в статусе `Draft`.
* **Постусловие:** Документ создан и доступен автору.

### UC-2: Управление версиями (Откат)
* **Действующее лицо:** Редактор (пользователь с ролью `EDITOR`) или пользователь с правом `WRITE`/`OWNER` в пространстве.
* **Предусловие:** Документ существует и имеет несколько версий.
* **Основной поток:**
    1. Редактор открывает документ.
    2. Переходит в раздел "История изменений".
    3. Выбирает две версии для сравнения.
    4. Система отображает визуальную разницу (diff).
    5. Редактор нажимает "Восстановить эту версию".
    6. Система создает новую версию документа, идентичную выбранной.
* **Постусловие:** Текущее состояние документа обновлено до выбранной версии.

### UC-3: Управление пространствами
* **Действующее лицо:** Администратор (пользователь с `is_admin = true`).
* **Предусловие:** Администратор авторизован.
* **Основной поток:**
    1. Администратор переходит в раздел "Управление пространствами".
    2. Нажимает "Создать пространство".
    3. Вводит название, описание и назначает ответственного редактора.
    4. Настраивает права доступа для пользователей и групп (Чтение/Запись).
    5. Сохраняет изменения.
* **Постусловие:** Новое пространство создано и готово к наполнению документами.

### UC-4: Назначение прав доступа
* **Действующее лицо:** Администратор (пользователь с `is_admin = true`).
* **Предусловие:** Администратор авторизован.
* **Основной поток:**
    1. Администратор открывает настройки пространства.
    2. Выбирает пользователя или группу из списка.
    3. Назначает уровень доступа (`READ`, `WRITE`, `OWNER`).
    4. Сохраняет изменения.
* **Постусловие:** Права доступа на пространство обновлены. Затронутые пользователи получают уведомления.

### UC-5: Управление группами пользователей
* **Действующее лицо:** Администратор (пользователь с `is_admin = true`).
* **Предусловие:** Администратор авторизован в системе.
* **Основной поток:**
    1. Администратор переходит в раздел "Группы пользователей" в панели администратора.
    2. Система отображает таблицу существующих групп (название, описание, количество участников).
    3. Администратор может:
        * **Создать группу:** Нажать "Создать группу" → ввести название и описание → сохранить. Система создает группу, группа отображается в таблице.
        * **Редактировать группу:** Нажать на группу → изменить название или описание → сохранить.
        * **Удалить группу:** Нажать "Удалить" → подтвердить действие. Система удаляет группу, все связи с пользователями и права группы на пространства удаляются каскадно. Пользователи не затрагиваются.
        * **Управлять составом:** Нажать на группу → перейти на страницу состава. Администратор видит список участников группы и выпадающий список всех пользователей системы. Может добавить пользователя в группу или исключить из неё.
* **Альтернативный поток (назначение прав группе):**
    1. Администратор переходит в раздел "Управление пространствами".
    2. Выбирает пространство → "Настройка прав".
    3. В интерфейсе назначения прав выбирает вкладку "Группы" (помимо вкладки "Пользователи").
    4. Выбирает группу из списка → назначает уровень доступа (`READ`, `WRITE`, `OWNER`) → сохраняет.
* **Постусловие:** Группы созданы/изменены. Права, полученные пользователями через группы, начинают действовать немедленно (при следующем запросе система вычисляет максимальное право).

### UC-6: Мягкое удаление пользователя
* **Действующее лицо:** Администратор (пользователь с `is_admin = true`).
* **Предусловие:** Администратор авторизован. Пользователь-цель активен (`is_deleted = false`).
* **Основной поток:**
    1. Администратор переходит в раздел "Управление пользователями" в панели администратора.
    2. Находит пользователя в таблице и нажимает "Удалить" (иконка корзины).
    3. Система открывает подтверждение: "Пользователь будет мягко удалён. Он потеряет доступ к системе, но его данные будут сохранены."
    4. Администратор подтверждает действие.
    5. Система устанавливает `is_deleted = true` для данного пользователя.
    6. Пользователь принудительно деаутентифицируется (если активные сессии есть).
    7. Система отображает подтверждение: "Пользователь <имя> мягко удалён".
* **Постусловие:** Пользователь не может войти в систему, исключён из списков и поиска. Его связанные записи (права на пространства, memberships в группах, авторство документов) сохраняются для истории и аудита.

### UC-7: Восстановление пользователя
* **Действующее лицо:** Администратор (пользователь с `is_admin = true`).
* **Предусловие:** Администратор авторизован. Пользователь-цель находится в состоянии мягкого удаления (`is_deleted = true`).
* **Основной поток:**
    1. Администратор переходит в раздел "Корзина" (или "Мягко удалённые пользователи") в панели администратора.
    2. Система отображает таблицу мягко удалённых пользователей (login, email, дата удаления).
    3. Администратор находит пользователя и нажимает "Восстановить".
    4. Система открывает подтверждение: "Пользователь будет восстановлен и получит доступ к системе с его первоначальной ролью".
    5. Администратор подтверждает действие.
    6. Система устанавливает `is_deleted = false` для данного пользователя.
    7. Система отображает подтверждение: "Пользователь <имя> восстановлен".
* **Постусловие:** Пользователь снова может войти в систему и получает права согласно своей роли (`GUEST`/`READER`/`EDITOR`), флага `is_admin` и назначенных прав на пространства.

### UC-8: Попытка удаления родительского документа
* **Действующее лицо:** Редактор (soft-delete) или Администратор (hard-delete).
* **Предусловие:** У документа есть один или несколько дочерних документов.
* **Основной поток:**
    1. Пользователь нажимает «Удалить» на странице документа.
    2. Система проверяет наличие дочерних документов:
        - Для soft-delete: ищет дочерние в статусах `Draft` или `Published`.
        - Для hard-delete: ищет дочерние в любом статусе, включая `Deleted`.
    3. Если найдены — система отклоняет действие.
    4. Система отображает сообщение: «Невозможно удалить документ: у него есть дочерние документы. Сначала удалите или переместите их.»
* **Постусловие:** Документ остаётся в прежнем статусе. Дочерние документы не затронуты.
