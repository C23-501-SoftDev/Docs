# US1.1.3 Валидация входящих данных при сохранении

## Описание
Реализация валидации данных на стороне бэкенда для обеспечения целостности и корректности сохраняемого контента документа.

## Технические требования
* **Стек/Инструменты:** Java/Spring Boot (Hibernate Validation/Bean Validation).
* **Архитектурные аспекты:**
    * Валидация DTO (название не пустое, ограничение длины).
    * Бизнес-валидация иерархии (защита от циклической зависимости, проверка принадлежности родителя к тому же пространству).
    * Проверка уникальности заголовка в рамках одного уровня вложенности.
    * Валидация ссылок на вложения (существование Attachment ID).
* **Инфраструктура/Конфигурация:** Обработка ошибок валидации и возврат соответствующих кодов ответов (400 Bad Request, 422 Unprocessable Entity).

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Валидация полей
* **Given:** Пользователь отправляет запрос на сохранение с некорректными данными (например, пустое название).
* **When:** Бэкенд выполняет валидацию.
* **Then:** Запрос отклоняется, система возвращает описание ошибки.

### Сценарий 2: Валидация иерархии
* **Given:** Пользователь пытается назначить родительским документом сам этот же документ или документ из другого пространства.
* **When:** Бэкенд выполняет бизнес-валидацию.
* **Then:** Запрос отклоняется с кодом 422, сохранение не производится.

### Сценарий 3: Успешная валидация
* **Given:** Пользователь отправляет запрос с корректно заполненными данными, прошедшими все структурные и логические проверки.
* **When:** Бэкенд выполняет валидацию.
* **Then:** Валидация пройдена, система сохраняет данные (оркестрация БД + Git).
