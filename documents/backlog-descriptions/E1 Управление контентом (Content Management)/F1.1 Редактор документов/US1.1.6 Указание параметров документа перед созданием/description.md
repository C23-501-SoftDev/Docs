# US1.1.6 Указание параметров документа перед созданием

## Описание
Реализация страницы и серверной логики для первого шага создания документа: выбор пространства (`Space`) и указание родительского документа (`Parent Document`) для корректного позиционирования новой страницы в иерархии Базы знаний.

## Технические требования
* **Стек/Инструменты:** Thymeleaf, Spring MVC, HTML Forms, `select` элементы.
* **Архитектурные аспекты:**
  * **Маршрут:** `GET /documents/new`. Доступен только ролям `Editor` и `Admin`.
  * **Модель данных:** На страницу передаются:
    1. Список доступных пользователю пространств (`List<Space>`) с правами на запись (`WRITE` или `OWNER`).
    2. Иерархический список документов в выбранном пространстве (для выбора родителя). Изначально может быть пустым или загружаться динамически через AJAX при смене пространства (опционально).
  * **Форма:** Отправка `POST /documents`.
    * Поле `spaceId` (обязательное).
    * Поле `parentDocumentId` (опциональное).
    * Поле `templateId` (скрытое или выбирается на следующем шаге, если это не US1.1.8).
  * **Валидация:** Сервер должен проверить, что пользователь действительно имеет право на запись в указанный `spaceId`.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Открытие формы создания
* **Given:** Редактор нажал кнопку «Создать документ».
* **When:** Загружается страница `/documents/new`.
* **Then:**
  1. Отображается выпадающий список «Пространство» (`select`), содержащий **только те пространства**, где у редактора есть права на создание документов.
  2. Если у редактора нет доступных пространств, список пуст и отображается сообщение «Обратитесь к администратору для назначения прав».

### Сценарий 2: Выбор родительского документа
* **Given:** Открыта форма создания.
* **When:** Пользователь выбирает пространство «Разработка» в первом списке.
* **Then:**
  1. Во втором списке «Родительский документ» (опционально) отображается иерархическое дерево документов пространства «Разработка» с отступами (например, `- Главная`, `-- Архитектура`, `-- API`).
  2. В списке отсутствуют документы в статусе `Deleted`.

### Сценарий 3: Успешное создание с параметрами
* **Given:** Пользователь выбрал `spaceId = 5`, `parentId = 12` и нажал «Продолжить».
* **When:** Отправлен POST-запрос.
* **Then:**
  1. Сервер создает новый `Document` со статусом `Draft`.
  2. В БД заполняются поля `space_id = 5` и `parent_document_id = 12`.
  3. Пользователь перенаправляется на страницу **редактора** `/documents/{newId}/edit` (см. US1.1.1) для ввода текста.

### Сценарий 4: Отказ в создании (Нарушение прав)
* **Given:** Злоумышленник вручную отправляет POST-запрос с `spaceId`, к которому у него нет доступа.
* **When:** Запрос обработан.
* **Then:** Сервер возвращает ошибку `403 Forbidden`. Документ не создается.
