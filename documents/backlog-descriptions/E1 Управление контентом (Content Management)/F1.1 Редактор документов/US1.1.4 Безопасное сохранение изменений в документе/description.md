# US1.1.4 Безопасное сохранение изменений в документе

## Описание
Реализация механизма безопасной отправки содержимого документа из Tiptap Editor на REST API сервера с защитой от CSRF-атак и корректной обработкой пользовательской сессии на основе HttpOnly Cookie.

## Технические требования
* **Стек/Инструменты:** JavaScript (Fetch API), Thymeleaf, Spring Security 6.
* **Архитектурные аспекты:**
  * **CSRF-защита:** Согласно `logic_scheme.md`, CSRF-токен передается в Cookie `XSRF-TOKEN`. Клиентский JS должен уметь читать эту Cookie и добавлять её значение в заголовок `X-XSRF-TOKEN` при каждом AJAX-запросе на изменение данных (POST, PUT, DELETE).
  * **Аутентификация:** JWT хранится в HttpOnly Cookie (`JWT`), поэтому JS не имеет к нему доступа и не должен добавлять заголовок `Authorization: Bearer ...`. Браузер автоматически прикрепит Cookie `JWT` к запросу, если установлен флаг `credentials: 'same-origin'`.
  * Обработка ответов сервера:
    * `200 OK`: Документ сохранен (показ уведомления об успехе).
    * `401 Unauthorized`: Cookie истекла или удалена — перенаправление на страницу логина (`window.location.href = '/login'`).
    * `403 Forbidden`: CSRF-токен невалиден или нет прав на редактирование.
    * `400 Bad Request`: Ошибки валидации (отображение ошибок в UI).

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Успешное сохранение с CSRF-токеном
* **Given:** Редактор открыт, пользователь внес правки.
* **When:** Нажата кнопка «Сохранить» (или Ctrl+S).
* **Then:**
  1. JS считывает Cookie `XSRF-TOKEN`.
  2. Отправляется `PUT /api/documents/{id}` с телом `{ "content": "...", "title": "..." }`.
  3. В заголовках запроса присутствует `X-XSRF-TOKEN: <token>`.
  4. Запрос включает `credentials: 'same-origin'` (для отправки Cookie `JWT`).
  5. После успешного ответа 200 пользователь видит некритичное уведомление «Документ сохранен».

### Сценарий 2: Обработка истечения сессии (401)
* **Given:** Редактор открыт, но время жизни Cookie `JWT` истекло.
* **When:** Нажата кнопка «Сохранить».
* **Then:**
  1. Сервер возвращает `401 Unauthorized`.
  2. Клиентский JS перенаправляет браузер на `/login` (без дополнительных диалогов, так как восстановить сессию может только повторный ввод пароля).

### Сценарий 3: Обработка CSRF-ошибки (403)
* **Given:** Cookie `XSRF-TOKEN` отсутствует (например, пользователь открыл старую вкладку после выхода).
* **When:** Нажата кнопка «Сохранить».
* **Then:**
  1. Сервер возвращает `403 Forbidden`.
  2. Пользователь видит сообщение: «Ошибка безопасности. Обновите страницу.».

### Сценарий 4: Автосохранение черновика
* **Given:** Редактор открыт в режиме создания нового документа или редактирования черновика.
* **When:** Пользователь прекращает печатать на 3 секунды.
* **Then:** Происходит автоматическая отправка PUT/POST запроса с теми же настройками безопасности (CSRF + Cookie).
