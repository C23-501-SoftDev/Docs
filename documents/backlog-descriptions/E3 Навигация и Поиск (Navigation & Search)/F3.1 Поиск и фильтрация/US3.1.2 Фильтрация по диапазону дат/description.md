# US3.1.2 Фильтрация по диапазону дат

## Описание
Реализация возможности фильтрации результатов поиска по дате создания или изменения документа для сужения выборки.

## Технические требования
* **Стек/Инструменты:** Java/Spring Boot, PostgreSQL.
* **Архитектурные аспекты:** Расширение API поиска параметрами `dateFrom` и `dateTo`. Использование условий SQL `BETWEEN` или операторов сравнения в запросах.
* **Инфраструктура/Конфигурация:** Поля `created_at` и `updated_at` в БД должны иметь индексы для быстрой фильтрации.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Успешная фильтрация
* **Given:** В системе есть документы, созданные/измененные в заданном диапазоне дат.
* **When:** Пользователь задает начальную и конечную даты и запускает фильтрацию.
* **Then:** Система возвращает список документов, соответствующих выбранным критериям времени.

### Сценарий 2: Невалидный диапазон дат
* **Given:** Пользователь задаёт `dateFrom` позже, чем `dateTo`.
* **When:** Пользователь запускает фильтрацию.
* **Then:** Система возвращает ошибку валидации с сообщением "Дата начала не может быть позже даты окончания". Запрос к БД не выполняется.

### Сценарий 3: Частичный фильтр (задана только одна дата)
* **Given:** Пользователь задаёт только `dateFrom` (без `dateTo`) или только `dateTo` (без `dateFrom`).
* **When:** Пользователь запускает фильтрацию.
* **Then:** Система применяет одностороннее ограничение: при наличии только `dateFrom` — возвращает все документы начиная с этой даты; при наличии только `dateTo` — все документы до этой даты.

### Сценарий 4: Корректность работы с часовыми поясами (UTC)
* **Given:** Пользователь находится в часовом поясе +3, документ создан в 10:00 UTC.
* **When:** Пользователь фильтрует по дате создания.
* **Then:** Система корректно сопоставляет дату, используя хранение и сравнение в UTC на стороне БД, независимо от локального времени клиента.

### Сценарий 5: Пустые параметры фильтрации
* **Given:** Пользователь очистил все поля фильтрации дат.
* **When:** Пользователь нажимает "Применить".
* **Then:** Фильтр сбрасывается, и система возвращает полный список документов (с учетом пагинации и прав доступа).

### Технические критерии готовности (Definition of Done)
* **Логирование:** В случае ошибки фильтрации или аномально долгого запроса, информация записывается в лог-файл (с указанием диапазона дат и ID пользователя).
* **Валидация:** Наличие unit-тестов, проверяющих граничные значения дат (будущие даты, некорректные форматы дат, пустые значения).
* **Безопасность (SQL Injection):** Верификация использования `PreparedStatement` для параметров `dateFrom` и `dateTo`.
* **API Документация:** Соответствие эндпоинта фильтрации требованиям Swagger, включая описание всех параметров запроса и кодов ответов (200, 400).
