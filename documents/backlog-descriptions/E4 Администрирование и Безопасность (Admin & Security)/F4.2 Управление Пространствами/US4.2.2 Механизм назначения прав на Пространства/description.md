# US4.2.2 Механизм назначения прав на Пространства

## Описание
Реализация функциональности управления доступом пользователей и групп к Пространствам, включая назначение прав (`READ`, `WRITE`, `OWNER`) для отдельных пользователей и групп в выбранном Space. Права на уровне пространства работают параллельно с глобальными ролями (`GUEST`, `READER`, `EDITOR`). Для пользователей с ролью `GUEST` это единственный способ получить доступ к контенту.

## Макет (Prototype)
[Открыть макет управления правами Пространства](../../../../prototypes/admin-panel/space-permissions.html) — прототип страницы назначения прав на пространство с таблицей текущих разрешений, формой добавления нового правила (пользователь/группа + уровень доступа READ/WRITE/OWNER) и возможностью отзыва прав.

### Взаимодействие глобальных ролей и прав пространства

| Глобальная роль | Права в пространстве | Итоговый доступ |
|----------------|---------------------|-----------------|
| `EDITOR` | Не требуются | Полный доступ ко всем документам |
| `READER` | Не требуются | Чтение всех документов |
| `GUEST` | `READ` | Чтение документов в этом пространстве |
| `GUEST` | `WRITE` | Создание и редактирование документов в этом пространстве |
| `GUEST` | `OWNER` | Полный доступ + управление правами в этом пространстве |
| `GUEST` | Отсутствуют | Нет доступа к пространству |

## Технические требования
* **Стек/Инструменты:** Java/Spring Boot (Security), PostgreSQL.
* **Архитектурные аспекты:**
  * Модель RBAC масштабирована на уровень отдельных пространств (иерархия доступа).
  * Таблицы-связки `space_user_permissions` (личные права) и `space_group_permissions` (права групп) с правами (`READ`, `WRITE`, `OWNER`).
  * Логика проверки доступа:
    1. Если глобальная роль `EDITOR` → доступ к операциям всегда разрешен.
    2. Если глобальная роль `READER` → разрешено только чтение.
    3. Если глобальная роль `GUEST` → проверяются записи в `space_user_permissions` и `space_group_permissions`.
    4. Для всех ролей: если `is_admin = true` → доступен раздел управления правами.
  * При назначении прав администратор выбирает пользователя или группу, пространство и тип права.
* **Инфраструктура/Конфигурация:**
  * API эндпоинты для управления правами доступны только администраторам (`is_admin = true`) и владельцам пространств (`OWNER`).
  * Эндпоинт: `POST /api/admin/spaces/{spaceId}/permissions` — назначение права пользователю или группе.
  * Эндпоинт: `DELETE /api/admin/spaces/{spaceId}/permissions/{permissionId}` — отзыв права.
  * Эндпоинт: `GET /api/admin/spaces/{spaceId}/permissions` — список прав пространства.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Успешное назначение права GUEST на чтение пространства
* **Given:** Администратор открывает настройки пространства.
* **When:** Назначает пользователю с ролью `GUEST` право `READ` на пространство.
* **Then:** 
  1. Сохраняется запись в `space_user_permissions` (space_id, user_id, permission_type='READ').
  2. Пользователь `GUEST` получает возможность видеть это пространство в списке доступных, просматривать список документов и читать их содержимое.

### Сценарий 2: Успешное назначение права GUEST на запись в пространстве
* **Given:** Администратор открывает настройки пространства.
* **When:** Назначает пользователю с ролью `GUEST` право `WRITE` на пространство.
* **Then:** 
  1. Сохраняется запись в `space_user_permissions` (permission_type='WRITE').
  2. Пользователь `GUEST` получает возможность создавать и редактировать документы в этом пространстве.

### Сценарий 3: Назначение права READER на пространство
* **Given:** Пользователь с ролью `READER` уже имеет глобальное право на чтение любых документов.
* **When:** Администратор назначает ему право `WRITE` на конкретное пространство.
* **Then:** Пользователь получает возможность редактировать документы в этом пространстве (поверх глобального права на чтение).

### Сценарий 4: Отзыв права у GUEST
* **Given:** Пользователь с ролью `GUEST` имеет право `READ` на пространство.
* **When:** Администратор отзывает это право (удаляет запись из `space_user_permissions`).
* **Then:** Пользователь теряет доступ к пространству и его документам.

### Сценарий 5: Отказ в доступе для GUEST без прав
* **Given:** Пользователь с ролью `GUEST` не имеет прав на пространство.
* **When:** Пытается получить список документов или открыть документ из этого пространства.
* **Then:** Система возвращает ошибку 403 Forbidden. Пространство не отображается в списке доступных.

### Сценарий 6: Назначение права группе на пространство
* **Given:** Администратор открывает настройки пространства и переходит на вкладку «Группы».
* **When:** Выбирает группу и назначает ей право `READ` на пространство.
* **Then:**
  1. Сохраняется запись в `space_group_permissions` (space_id, group_id, permission_type='READ').
  2. Все участники группы получают доступ к пространству в соответствии с назначенным правом.
