# US4.1.2 Создание сущности User и JWT авторизация

## Описание
Реализация подсистемы идентификации пользователей. Обеспечивает аутентификацию по паре "логин/пароль" с выдачей JWT-токена, который сохраняется в **HttpOnly Cookie** браузера и используется для бесстатусной (stateless) авторизации во всех последующих запросах к MVC-контроллерам и REST API.

## Макет (Prototype)
[Открыть макет JWT аутентификации](../../../../prototypes/admin-panel/auth-setup.html) — прототип с симуляцией логина/логаута, отображением JWT-токена и формой настройки секретного ключа и времени жизни токена.

## Технические требования
* **Стек/Инструменты:** Java 17+, Spring Boot, Spring Security, JWT (jjwt), Spring Data JPA, BCrypt, PostgreSQL 18.
* **Архитектурные аспекты:**
  * Реализация сущности `User` в точном соответствии со схемой данных `120.data-model.md` (Таблица `users`):
    * `id`, `login`, `password_hash`, `email`, `role` (`Admin`, `Editor`, `Reader`), `created_at`, `updated_at`.
  * Интеграция подсистемы **Issuer** (генерация JWT при логине) и **Validator** (проверка JWT, извлеченного из Cookie, при каждом запросе).
  * **Stateless Sessions:** `SessionCreationPolicy.STATELESS` — сервер не хранит сессию.
  * Хранение паролей с использованием алгоритма BCrypt (соответствует полю `password_hash` в БД).
  * **JWT состав:** В токен включается claim `role` со значением роли пользователя (см. `070.role-matrix.md`). При валидации токена Spring Security извлекает роль из JWT для принятия решений об авторизации.
* **Инфраструктура/Конфигурация:**
  * Настройка Spring Security для защиты REST API и MVC эндпоинтов.
  * Конфигурация параметров JWT в `application.properties` (секретный ключ, время жизни токена).
  * Эндпоинт для входа: `POST /api/login` (принимает JSON с `login`/`password`, возвращает 200 OK и устанавливает Cookie `JWT`).
  * Эндпоинт для выхода: `POST /api/logout` (очищает Cookie `JWT`).

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Успешная аутентификация (Выдача Cookie)
* **Given:** Пользователь зарегистрирован в таблице `users` БД PostgreSQL.
* **When:** Отправлен POST-запрос на `/api/login` с корректными `login` и `password`.
* **Then:** 
  1. Система сверяет пароль с `password_hash` (BCrypt).
  2. Система генерирует JWT с claim `role`.
  3. Система возвращает ответ `200 OK`.
  4. В ответе присутствует заголовок `Set-Cookie` с именем `JWT`, значением `<token>`, флагами `HttpOnly` (защита от XSS) и `Secure` (для HTTPS).

### Сценарий 2: Отказ при неверных учетных данных
* **Given:** Пользователь ввел неверный пароль или несуществующий логин.
* **When:** Отправлен запрос на `/api/login`.
* **Then:** Система возвращает ошибку `401 Unauthorized`. Cookie `JWT` не устанавливается.

### Сценарий 3: Безопасность данных (Хеширование)
* **Given:** Создание новой учетной записи (через админ-панель US4.1.4).
* **When:** Пароль передан на сервер.
* **Then:** Пароль сохраняется в поле `password_hash` таблицы `users` строго в зашифрованном виде (BCrypt). Открытый пароль не должен фигурировать в БД или логах.

### Сценарий 4: Валидация JWT из Cookie при защищенном запросе
* **Given:** Пользователь получил валидный JWT-токен и Cookie `JWT` установлена в браузере.
* **When:** Выполняется запрос к защищенному ресурсу (например, `GET /documents/1`).
  * Браузер автоматически прикрепляет Cookie `JWT`.
* **Then:** 
  1. Фильтр `JwtCookieAuthenticationFilter` извлекает токен из Cookie.
  2. Система успешно валидирует токен (подпись, срок действия).
  3. Система извлекает роль из claim `role`.
  4. Запрос выполняется в соответствии с правами роли (см. `070.role-matrix.md`).

### Сценарий 5: Истекший токен
* **Given:** JWT-токен в Cookie `JWT` истек (превышено время жизни `expiration`).
* **When:** Выполняется запрос к защищенному ресурсу.
* **Then:** Система отклоняет запрос, возвращая ошибку `401 Unauthorized`.

### Сценарий 6: Отсутствие токена
* **Given:** Пользователь не авторизован (Cookie `JWT` отсутствует в запросе).
* **When:** Выполняется запрос к защищенному ресурсу.
* **Then:** Система возвращает ошибку `401 Unauthorized`.

### Сценарий 7: Завершение сессии (Logout)
* **Given:** Пользователь авторизован.
* **When:** Отправлен POST-запрос на `/api/logout`.
* **Then:** Сервер отправляет заголовок `Set-Cookie`, который перезаписывает `JWT` пустым значением с истекшим сроком действия (`Max-Age=0`), тем самым удаляя токен на клиенте.
