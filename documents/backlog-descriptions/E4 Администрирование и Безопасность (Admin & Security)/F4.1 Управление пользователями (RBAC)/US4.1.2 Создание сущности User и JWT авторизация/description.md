# US4.1.2 Создание сущности User и JWT авторизация

## Описание
Реализация подсистемы идентификации пользователей. Обеспечивает аутентификацию по паре "логин/пароль" с выдачей JWT-токена, который используется для авторизации во всех последующих запросах.

## Технические требования
* **Стек/Инструменты:** Java 17+, Spring Boot, Spring Security, JWT (jjwt), Spring Data JPA, BCrypt, PostgreSQL 18.
* **Архитектурные аспекты:**
  * Реализация сущности `User` с полями: `id`, `login`, `password_hash`, `email`, `role`, `created_at`, `updated_at`.
  * Интеграция подсистемы **Issuer** (генерация JWT) и **Validator** (проверка JWT при каждом запросе).
  * Хранение паролей с использованием алгоритма BCrypt.
  * **JWT состав:** При генерации JWT в токен включается claim `role` со значением роли пользователя (`Admin`, `Editor`, `Reader`). При валидации токена Spring Security извлекает роль из JWT для принятия решений об авторизации.
* **Инфраструктура/Конфигурация:**
  * Настройка Spring Security для защиты REST API.
  * Конфигурация параметров JWT в `application.properties` (секретный ключ, время жизни токена).
  * Схема таблицы `users` определена в `readme.md`.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Успешная аутентификация
* **Given:** Пользователь зарегистрирован в системе (логин и пароль существуют в БД).
* **When:** Отправлен POST-запрос на `/api/login` с корректными логином и паролем.
* **Then:** Система возвращает JWT-токен, который должен быть передан в заголовке `Authorization: Bearer <token>` при последующих запросах.

### Сценарий 2: Отказ при неверных учетных данных
* **Given:** Пользователь ввел неверный пароль или логин.
* **When:** Отправлен запрос на `/api/login`.
* **Then:** Система возвращает ошибку 401 Unauthorized, токен не выдается.

### Сценарий 3: Безопасность данных
* **Given:** Создание новой учетной записи (через админ-панель).
* **When:** Пароль передан на сервер.
* **Then:** Пароль сохраняется в БД строго в зашифрованном виде (BCrypt), никогда не хранится в открытом виде.

### Сценарий 4: Валидация JWT при защищенном запросе
* **Given:** Пользователь получил валидный JWT-токен.
* **When:** Выполняется запрос к защищенному ресурсу с заголовком `Authorization: Bearer <token>`.
* **Then:** Система успешно валидирует токен и выполняет запрос.

### Сценарий 5: Истекший токен
* **Given:** JWT-токен истек (превышено время жизни).
* **When:** Выполняется запрос к защищенному ресурсу с истекшим токеном.
* **Then:** Система отклоняет запрос, возвращая ошибку 401 Unauthorized.

### Сценарий 6: Отсутствие токена
* **Given:** Пользователь не авторизован (JWT отсутствует в заголовке `Authorization`).
* **When:** Выполняется запрос к защищенному ресурсу.
* **Then:** Система возвращает ошибку 401 Unauthorized.
