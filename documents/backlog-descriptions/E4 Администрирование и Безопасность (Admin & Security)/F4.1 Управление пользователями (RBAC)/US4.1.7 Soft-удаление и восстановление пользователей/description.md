# US4.1.7 Soft-удаление и восстановление пользователей

## Описание
Реализация процесса удаления пользователей через пометку "неактивный" (Soft-delete), позволяющего скрывать пользователей из общих списков и предотвращать их авторизацию, но сохранять данные в системе для целей аудита и возможности восстановления.

## Технические требования
* **Стек/Инструменты:** Java/Spring Boot (Backend), PostgreSQL 18.
* **Архитектурные аспекты:** Реализация логического флага `active` (boolean) в сущности `User`. Обновление сервисов аутентификации (блокировка входа, если `active=false`) и запросов списка пользователей (исключение неактивных по умолчанию).
* **Инфраструктура/Конфигурация:** Настройка прав доступа для выполнения Soft-delete пользователю с ролью Администратор.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Soft-delete пользователя (делаем Inactive)
* **Given:** Пользователь находится в статусе "Active".
* **When:** Администратор выполняет операцию блокировки/удаления (Soft-delete).
* **Then:** 
    * Пользователь получает статус "Inactive".
    * Пользователь не может войти в систему (авторизация блокируется).
    * Пользователь исчезает из основных списков активных пользователей.

### Сценарий 2: Восстановление пользователя (делаем Active)
* **Given:** Пользователь имеет статус "Inactive".
* **When:** Администратор выполняет операцию восстановления.
* **Then:** 
    * Пользователь возвращается в статус "Active".
    * Пользователь снова может авторизоваться в системе.
    * Пользователь снова появляется в списках активных пользователей.
