# US4.1.6 Управление списком пользователей через таблицу

## Описание
Разработка страницы административной панели для управления учетными записями пользователей системы. Интерфейс должен представлять данные в виде таблицы с возможностью фильтрации, а также предоставлять элементы управления для добавления, блокировки (смены роли) и редактирования пользователей.

## Макет (Prototype)
[Открыть макет управления списком пользователей](../../../../prototypes/admin-panel/users.html) — прототип с сортируемой таблицей, поиском, фильтрацией по ролям и модальными окнами.

## Технические требования
* **Стек/Инструменты:** Thymeleaf, Spring MVC, HTML Forms, CSS (Tables).
* **Архитектурные аспекты:**
  * **Маршрут:** `GET /admin/users`. Доступен только роли `ROLE_ADMIN`.
  * **Серверный рендеринг:** Контроллер получает список всех пользователей из `UserRepository`, добавляет его в `Model` и передает на рендеринг в Thymeleaf.
  * **Структура страницы:**
    * Наследует общий `layout.html` (см. US3.3.1).
    * **Заголовок:** «Управление пользователями».
    * **Кнопка действия:** «Добавить пользователя» (ведет на `GET /admin/users/new` или открывает модальное окно с формой).
    * **Поле фильтрации:** Текстовое поле для фильтрации таблицы по логину или email на стороне клиента (JavaScript) или отправка формы на сервер (Thymeleaf + параметры запроса).
    * **Таблица пользователей:**
      * Колонки: `ID`, `Логин`, `Email`, `Роль`, `Дата создания`, `Действия`.
      * Сортировка по колонкам (опционально, может быть реализовано через клик по заголовку).
  * **Управление ролями (RBAC):**
    * В столбце «Роль» отображается текущая роль (`Admin`, `Editor`, `Reader`).
    * Выпадающий список или кнопки для быстрой смены роли пользователя (отправка PUT/POST запроса на `/api/admin/users/{id}/role`).
    * **Валидация:** Администратор не может понизить роль самому себе (чтобы не потерять доступ к панели).
  * **Блокировка пользователя:**
    * Согласно `050.entities.md`, физического удаления или поля `blocked` нет. Блокировка осуществляется сменой роли на `Reader` (или специальную заглушку) или удалением записи.
    * Кнопка «Заблокировать» меняет роль пользователя на `Reader` (если это не администратор).
* **Инфраструктура/Конфигурация:**
  * Использование `spring-boot-starter-validation` для проверки email и логина при создании/редактировании.
  * Обработка ошибок: Отображение сообщений об ошибках валидации (например, «Пользователь с таким email уже существует»).

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Просмотр списка пользователей
* **Given:** Администратор авторизован.
* **When:** Переход на страницу `/admin/users`.
* **Then:**
  1. Отображается таблица со всеми пользователями, зарегистрированными в таблице `users`.
  2. Для каждого пользователя виден `ID`, `Логин`, `Email`, `Роль`.
  3. Пароли (`password_hash`) в таблице не отображаются (скрыты или показаны звездочками).

### Сценарий 2: Фильтрация таблицы
* **Given:** На странице отображается 50 пользователей.
* **When:** Администратор вводит текст в поле «Поиск по логину или email».
* **Then:** Таблица динамически обновляется (через JS-фильтрацию DOM или через AJAX-подгрузку), показывая только строки, содержащие введенный текст.

### Сценарий 3: Смена роли пользователя
* **Given:** В таблице есть пользователь `editor_ivan` с ролью `Editor`.
* **When:** Администратор в строке этого пользователя выбирает в выпадающем списке роль `Reader` и нажимает «Применить».
* **Then:**
  1. Отправляется запрос на обновление роли.
  2. В БД поле `role` обновляется на `Reader`.
  3. Таблица перерисовывается: в строке `editor_ivan` роль изменяется на `Reader`.

### Сценарий 4: Защита от понижения собственной роли
* **Given:** Администратор `admin_root` зашел в панель.
* **When:** Пытается изменить роль самому себе на `Editor` или `Reader`.
* **Then:**
  1. Система отклоняет запрос с ошибкой `400 Bad Request`.
  2. Отображается сообщение: «Нельзя изменить собственную роль администратора».

### Сценарий 5: Создание нового пользователя
* **Given:** Нажата кнопка «Добавить пользователя».
* **When:** Заполнена форма с полями `Логин`, `Email`, `Пароль`, `Роль` и нажато «Сохранить».
* **Then:**
  1. Данные проходят валидацию (уникальность логина/email, сложность пароля).
  2. Создается новая запись в таблице `users`.
  3. Пароль сохраняется в виде BCrypt-хэша в поле `password_hash`.
  4. Администратор видит нового пользователя в таблице.

### Сценарий 6: Доступ запрещен (403)
* **Given:** Пользователь с ролью `Editor` пытается открыть URL `/admin/users`.
* **When:** Запрос отправлен.
* **Then:** Spring Security возвращает ошибку `403 Forbidden` (или перенаправляет на страницу с ошибкой доступа).
