# US4.1.3 Реализация RBAC

## Описание
Внедрение логики разграничения прав доступа (Role-Based Access Control) на прикладном уровне. Система определяет, какие действия (Read, Create, Edit, Delete) доступны пользователю на основе его **глобальной роли** (`GUEST`, `READER`, `EDITOR`), **флага администратора** (`is_admin`) и **прав на уровне пространства (Space)**.

### Совместная работа глобальной роли, флага is_admin и прав пространства

| Глобальная роль | is_admin | Смысл |
|----------------|:--------:|-------|
| **GUEST** | `false` | Может только войти в систему. Все права на контент — через `SpacePermission` и `DocumentPermission` |
| **GUEST** | `true` | «Чистый администратор» — управляет пользователями и правами, но не имеет автоматических прав на контент |
| **READER** | `false` | Может читать любые документы в системе |
| **READER** | `true` | Читает любые документы + имеет доступ к панели администратора |
| **EDITOR** | `false` | Может создавать, редактировать и удалять любые документы в системе |
| **EDITOR** | `true` | Полный доступ: к контенту (CRUD) и к администрированию |

## Технические требования
* **Стек/Инструменты:** Java 17+, Spring Boot, Spring Security, JWT, Spring Data JPA.
* **Архитектурные аспекты:**
  * Реализация политик безопасности, ограничивающих доступ к API методам в зависимости от:
    - Глобальной роли пользователя (`GUEST`, `READER`, `EDITOR`)
    - Флага `is_admin`
    - Прав на уровне пространства (`space_permissions` с типами `READ`, `WRITE`, `OWNER`)
  * Логика проверки доступа к операциям с документами:
    - `EDITOR` → всегда разрешено (Create/Read/Edit/Delete)
    - `READER` → разрешено только Read
    - `GUEST` → разрешено только при наличии соответствующих прав в `space_permissions` или `document_permissions`
    - `is_admin = true` → всегда разрешен доступ к `/admin/**`, не влияет на права к контенту
  * Чтение глобальной роли и флага `is_admin` из JWT-токена (claims `role` и `is_admin`).
  * Реализация эндпоинта `/api/user/permissions?spaceId={id}` для получения прав пользователя в пространстве (используется фронтендом для UI-интеграции).
* **Инфраструктура/Конфигурация:**
  * Настройка Spring Security с использованием `@PreAuthorize` или аналогичных механизмов.
  * Таблица `space_permissions` должна быть заполнена тестовыми данными для проверки сценариев.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Редактирование документа редактором
* **Given:** Пользователь обладает глобальной ролью `EDITOR`.
* **When:** Он пытается выполнить операцию создания (Create), редактирования (Edit) или удаления (Delete) документа.
* **Then:** Система разрешает выполнение операции (независимо от прав на пространство).

### Сценарий 2: Чтение документа читателем
* **Given:** Пользователь обладает глобальной ролью `READER`.
* **When:** Он запрашивает просмотр любого документа.
* **Then:** Система успешно возвращает содержимое документа.

### Сценарий 3: Отказ в редактировании для читателя
* **Given:** Пользователь обладает глобальной ролью `READER`.
* **When:** Он пытается создать, отредактировать или удалить документ.
* **Then:** Система возвращает ошибку 403 Forbidden.

### Сценарий 4: Доступ GUEST через права на пространство
* **Given:** Пользователь обладает глобальной ролью `GUEST` и имеет право `READ` в пространстве (запись в `space_permissions`).
* **When:** Он запрашивает просмотр документа в этом пространстве.
* **Then:** Система успешно возвращает содержимое документа.

### Сценарий 5: Отказ GUEST без прав на пространство
* **Given:** Пользователь обладает глобальной ролью `GUEST` и **не имеет** прав в пространстве.
* **When:** Он пытается просмотреть документ в этом пространстве.
* **Then:** Система возвращает ошибку 403 Forbidden.

### Сценарий 6: Административный доступ
* **Given:** Пользователь обладает флагом `is_admin = true`.
* **When:** Он обращается к API управления пользователями (`/api/admin/*`) или пространствами.
* **Then:** Система предоставляет доступ к административным функциям. Права на контент определяются глобальной ролью.

### Сценарий 7: Безопасность в пользовательском интерфейсе
* **Given:** Пользователь заходит на страницу просмотра документа.
* **Then:** Система возвращает клиенту не только содержимое документа, но и флаги прав (`canEdit`, `canDelete`, `canCreate`) через эндпоинт `/api/user/permissions` или в теле ответа документа.
* **And:** Фронтенд скрывает компоненты управления (кнопки «Редактировать», «Удалить», «Создать») от пользователей, у которых соответствующие флаги равны `false`.
