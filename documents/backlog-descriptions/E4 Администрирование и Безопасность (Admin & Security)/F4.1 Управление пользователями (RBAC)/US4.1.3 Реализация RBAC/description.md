# US4.1.3 Реализация RBAC

## Описание
Внедрение логики разграничения прав доступа (Role-Based Access Control) на прикладном уровне. Система определяет, какие действия (Read, Create, Edit, Delete) доступны пользователю на основе его **глобальной роли** и **прав на уровне пространства (Space)**.

### Совместная работа глобальной роли и прав пространства

| Глобальная роль | Смысл |
|----------------|-------|
| **Reader** | «Только читатель» — никогда не может создавать или редактировать документы, даже если ошибочно получит WRITE в пространстве |
| **Editor** | «Потенциальный редактор» — может создавать и редактировать документы, но **только в тех пространствах**, где ему выдано право WRITE или OWNER |
| **Admin** | «Всемогущий» — имеет неограниченный доступ ко всем пространствам и документам, а также может управлять пользователями и пространствами |

## Технические требования
* **Стек/Инструменты:** Java 17+, Spring Boot, Spring Security, JWT, Spring Data JPA.
* **Архитектурные аспекты:**
  * Реализация политик безопасности, ограничивающих доступ к API методам в зависимости от:
    - Глобальной роли пользователя (Admin, Editor, Reader)
    - Прав на уровне пространства (`space_permissions` с типами READ, WRITE, OWNER)
  * Логика проверки доступа к операциям с документами:
    - `Admin` → всегда разрешено
    - `Reader` → всегда запрещено для операций Create/Edit/Delete
    - `Editor` → разрешено только при наличии WRITE или OWNER в соответствующем пространстве
  * Чтение глобальной роли из JWT-токена (claim `role`).
  * Реализация эндпоинта `/api/user/permissions?spaceId={id}` для получения прав пользователя в пространстве (используется фронтендом для UI-интеграции).
* **Инфраструктура/Конфигурация:**
  * Настройка Spring Security с использованием `@PreAuthorize` или аналогичных механизмов.
  * Таблица `space_permissions` должна быть заполнена тестовыми данными для проверки сценариев.

## Критерии приемки (Acceptance Criteria)

### Сценарий 1: Редактирование документа редактором с правами на пространство
* **Given:** Пользователь обладает глобальной ролью `Editor` **и имеет право WRITE или OWNER в данном пространстве**.
* **When:** Он пытается выполнить операцию создания (Create), редактирования (Edit) или удаления (Delete) документа.
* **Then:** Система разрешает выполнение операции.

### Сценарий 2: Отказ в редактировании для редактора без прав на пространство
* **Given:** Пользователь обладает глобальной ролью `Editor`, но **не имеет права WRITE или OWNER в данном пространстве** (отсутствует запись в `space_permissions` с типом WRITE или OWNER).
* **When:** Он пытается создать, отредактировать или удалить документ в этом пространстве.
* **Then:** Система возвращает ошибку 403 Forbidden.

### Сценарий 3: Ограничение прав для пользователя со статусом Reader
* **Given:** Пользователь обладает глобальной ролью `Reader` (независимо от наличия прав WRITE в пространстве).
* **When:** Он пытается выполнить операции создания (Create), редактирования (Edit) или удаления (Delete) документа.
* **Then:** Система отклоняет запрос, возвращая ошибку 403 Forbidden.

### Сценарий 4: Доступ на чтение для Reader
* **Given:** Пользователь обладает глобальной ролью `Reader` и имеет право READ в пространстве (или пространство публичное).
* **When:** Он запрашивает просмотр документа.
* **Then:** Система успешно возвращает содержимое документа.

### Сценарий 5: Административный доступ
* **Given:** Пользователь обладает глобальной ролью `Admin`.
* **When:** Он обращается к API управления пользователями или пространствами, а также к любым документам в любых пространствах.
* **Then:** Система предоставляет полный доступ ко всем функциям без дополнительных проверок прав на пространство.

### Сценарий 6: Безопасность в пользовательском интерфейсе
* **Given:** Пользователь заходит на страницу просмотра документа.
* **Then:** Система возвращает клиенту не только содержимое документа, но и флаги прав (`canEdit`, `canDelete`, `canCreate`) через эндпоинт `/api/user/permissions` или в теле ответа документа.
* **And:** Фронтенд скрывает компоненты управления (кнопки «Редактировать», «Удалить», «Создать») от пользователей, у которых соответствующие флаги равны `false`.
