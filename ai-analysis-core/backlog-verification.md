# Backlog verification

В данном файле описано как нужно проводить верификацию и трассировку требований в проекте

## Предусловия

Файл documents\backlog.json создан на основании файлов (правила формирования описаны ai-analysis-core\backlog-structure.md):
base-requirements.md
logic_scheme.md

## Проверка бэклога на полноту

Для обеспечения полноты и обоснованности бэклога, каждый элемент (Epic, Feature, Task) должен содержать заполненный блок `requirements_check` в метаданных. Это позволяет подтвердить, что каждая запись в бэклоге продиктована конкретными требованиями бизнеса или технической логикой.

### Правила заполнения `requirements_check`

При формировании или обновлении `backlog.json` необходимо следовать следующим правилам:

1. **Идентификация источника (`document`)**:
    * Указывается относительный путь к файлу или точное название документа, из которого взято требование (например, `base-requirements.md` или `logic_scheme.md`).
    * Если требование вытекает из нескольких документов, создается несколько объектов в массиве `requirements_check`.

2. **Обоснование (`texts`)**:
    * В массив `texts` копируются прямые цитаты, ключевые фразы или описания логических узлов из документа-источника.
    * Цитаты должны однозначно подтверждать необходимость реализации данного пункта бэклога.
    * Для технических задач, вытекающих из архитектурных решений, в качестве текста может быть указано описание конкретного шага процесса из `logic_scheme.md`.

### Пример заполнения в JSON

```json
"metadata": {
  "requirements_check": [
    {
      "document": "base-requirements.md",
      "texts": [
        "Система должна обеспечивать автоматическую проверку структуры бэклога",
        "Результат проверки должен сохраняться в формате JSON"
      ]
    }
  ]
}
```
