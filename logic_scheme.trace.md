# Логическая схема системы

В данном файле представлена логическая схема системы, описанной в [base-requirements.md](base-requirements.md)

## 1. Общая структура

### 1.1. Фронтенд (Клиентская часть - Thymeleaf + JavaScript)

- Предоставляет пользовательский интерфейс: просмотр и редактирование Wiki-страниц, управление справочником пользователей, поиск.
- Реализован с использованием **серверного рендеринга (SSR)** на основе шаблонизатора **Thymeleaf**.
[
   - **Требование:** Thymeleaf SSR
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.5, US3.3.1]
]
- Страницы формируются на сервере и отправляются клиенту в виде готового HTML.
- Интерактивные элементы реализованы с помощью встроенного JavaScript.
- Включает встраиваемый **Tiptap Editor** – WYSIWYG-редактор с поддержкой Markdown и двухпанельным интерфейсом (исходный код/результат), подключаемый как JavaScript-библиотека на страницах редактирования.
[
   - **Требование:** Tiptap WYSIWYG-редактор
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.1]
]
- Данные для инициализации редактора встраиваются непосредственно в HTML-страницу сервером через `window.__INITIAL_CONTENT__`.
- Для асинхронного сохранения изменений редактор взаимодействует с REST API через **Fetch API**.
[
   - **Требование:** AJAX сохранение через Fetch API
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.4]
]
- Аутентификация реализована через **JWT-токен, хранящийся в HttpOnly Cookie**, что обеспечивает защиту от XSS-атак и автоматическую отправку токена с каждым запросом.
[
   - **Требование:** JWT в HttpOnly Cookie
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.2]
]
- Для защиты от CSRF-атак используется CSRF-токен, передаваемый в Cookie и добавляемый JavaScript в заголовок `X-XSRF-TOKEN` при AJAX-запросах.
[
   - **Требование:** CSRF защита
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.4]
]
- Навигация между страницами осуществляется через обычные HTTP-переходы (ссылки и формы), что исключает необходимость в клиентском роутинге.
[
   - **Требование:** Серверная навигация
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US3.3.1]
]

### 1.2. Бэкенд (Backend – Spring Boot)

Реализован на **Java** с использованием **Spring Boot**, сборка – **Maven**.
[
   - **Требование:** Spring Boot Backend
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.1]
]

#### 1.2.1. Spring MVC

- Формирует и отдаёт на фронтенд HTML-страницы (View) с помощью шаблонизатора **Thymeleaf**.
[
   - **Требование:** Spring MVC + Thymeleaf
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.5, US3.3.1]
]
- Обрабатывает HTTP-запросы, маршрутизацию.
- Отвечает за передачу статических ресурсов (CSS, JS).
- Встраивает данные документов и пользователей непосредственно в генерируемые HTML-страницы через модель (`Model`).
- Контроллеры (`@Controller`) возвращают имена представлений (View), которые рендерятся на сервере.

#### 1.2.2. Spring Security + JWT

- Обеспечивает аутентификацию и авторизацию в **stateless** режиме (`SessionCreationPolicy.STATELESS`).
[
   - **Требование:** Stateless JWT Authentication
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.2, US4.1.3]
]
- При логине проверяет учётные данные (логин + хеш пароля) через БД и выдаёт JWT-токен.
- JWT-токен передаётся клиенту и хранится в **HttpOnly Cookie** (имя Cookie: `JWT`), что защищает от XSS-атак.
- Для каждого запроса сервер выполняет валидацию JWT-токена, извлекая его из Cookie (фильтр `JwtCookieAuthenticationFilter`).
- Сервер **не хранит состояние сессии**, что обеспечивает горизонтальную масштабируемость.
- Для защиты от CSRF-атак генерируется CSRF-токен, который передаётся клиенту в Cookie `XSRF-TOKEN` (доступной для чтения JavaScript) и проверяется сервером в заголовке `X-XSRF-TOKEN` при state-changing запросах.
- MVC-контроллеры и REST-контроллеры используют единый механизм аутентификации через JWT из Cookie.

#### 1.2.3. REST API

- Контроллеры (`@RestController`) и DTO для обмена данными.
- Принимает AJAX-запросы от клиентского JavaScript (в основном от Tiptap Editor для сохранения документов, загрузки вложений).
[
   - **Требование:** REST API для редактора
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.4, US1.4.1]
]
- Возвращает JSON или статусы HTTP.
- Взаимодействует с бизнес-логикой.
- Аутентификация для REST-эндпоинтов также основана на JWT из Cookie (проверяется тем же `JwtCookieAuthenticationFilter`).

#### 1.2.4. Чистая архитектура (Clean Architecture)

Архитектура построена по принципу Clean Architecture (или аналогичному), с явным разделением на слои:
[
   - **Требование:** Clean Architecture
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.1]
]
- **Domain (ядро):** Сущности (`Document`, `Space`, `User`), интерфейсы репозиториев (`DocumentRepository`, `SpaceRepository`). Не зависит от Spring и фреймворков.
- **Use Cases:** Сценарии использования (`CreateDocumentUseCase`, `UpdateDocumentUseCase`, `GetDocumentHistoryUseCase`). Содержат бизнес-логику, работают через интерфейсы репозиториев.
- **Interface (адаптеры):** REST Controllers, Presenters. Преобразуют запросы/ответы между внешним миром и Use Cases.
- **Infrastructure:** Реализации репозиториев (Spring Data JPA, JGit), отправка email, конфигурации.

#### 1.2.5. JGit (Git операции)

- Выполняет низкоуровневые Git-операции (коммиты, просмотр истории, откат, ветвление).
- Работает с Git-репозиторием, где хранятся `.md`-файлы Wiki.
[
   - **Требование:** JGit интеграция
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US2.2.5]
]

#### 1.2.6. Async Email

- Асинхронная отправка уведомлений (например, при изменении страницы, регистрации).
- Взаимодействует с внешним SMTP-сервером (Mail.ru).
[
   - **Требование:** Асинхронные Email-уведомления
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.3.1]
]

#### 1.2.7. Repository (Spring Data JPA)

- Слой доступа к реляционной базе данных.
- CRUD-операции над сущностями (пользователи, метаданные Git, теги, категории, настройки).
[
   - **Требование:** Spring Data JPA
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.1, US1.1.2, US4.1.4]
]

- Хранит метаданные о версиях (хэши коммитов, авторы, даты, сообщения), индексы для поиска, справочник пользователей, права доступа, теги, категории и вспомогательные данные.
[
   - **Требование:** PostgreSQL метаданные
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.4.2, US2.2.2, US4.1.1]
]

#### 1.3.2. Git репозиторий

- Физическое хранилище Markdown-файлов (`.md`).
- Каждый документ – отдельный файл.
- Полное версионирование через Git (история, откат, сравнение версий).
[
   - **Требование:** Git-версионность
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US2.2.1, US2.2.5]
]

#### 1.3.3. Blob хранилище

- Ссылки на blob-объекты хранятся в метаданных PostgreSQL.
[
   - **Требование:** Blob хранилище
   - **Статус:** Покрыто
   - **Элементы бэклога:** [F1.4, US1.4.1, US1.4.2]

[
   - **Требование:** Интеграция с SMTP

   - **Элементы бэклога:** [US4.3.1]

## 2. Потоки данных

1. **Инициализация** – пользователь заходит на сайт → сервер через Spring MVC формирует и отдаёт HTML-страницу (View) с шаблонизатором Thymeleaf, содержащую весь необходимый контент и минимальный JavaScript для интерактивности.
[
   - **Требование:** Поток инициализации страницы
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US3.3.1]
]

2. **Аутентификация** – пользователь вводит логин и пароль → фронтенд отправляет их через HTML-форму на `/login` → Spring Security проверяет данные через БД → формирует JWT → устанавливает JWT в HttpOnly Cookie `JWT` → устанавливает CSRF-токен в Cookie `XSRF-TOKEN` (доступную для JavaScript) → выполняет редирект на главную страницу.
[
   - **Требование:** Процесс аутентификации (JWT в Cookie)
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.2]
]

3. **Защищённые запросы страниц** – при переходе по ссылкам и отправке форм браузер автоматически отправляет Cookie `JWT` с каждым запросом → сервер через `JwtCookieAuthenticationFilter` извлекает JWT из Cookie и валидирует его → при успехе Spring MVC формирует HTML-страницу с данными, иначе – редирект на страницу логина.
[
   - **Требование:** Защита страниц через Cookie JWT
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.2, US4.1.3]
]

4. **Защищённые AJAX-запросы** – клиентский JavaScript (Tiptap Editor) читает CSRF-токен из Cookie `XSRF-TOKEN` → отправляет AJAX-запрос к REST API с заголовком `X-XSRF-TOKEN` и автоматически прикреплённой Cookie `JWT` → сервер валидирует JWT из Cookie и CSRF-токен из заголовка → при успехе выполняет бизнес-логику и возвращает JSON, иначе – 401/403.
[
   - **Требование:** AJAX с CSRF и JWT Cookie
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.4]
]

5. **Получение Wiki-страницы** – клиент запрашивает `/documents/{id}` → MVC-контроллер вызывает `DocumentService` → сервис через JGit читает соответствующий `.md`-файл из Git-репозитория → получает метаданные из PostgreSQL → контроллер добавляет данные в модель (`Model`) → Thymeleaf рендерит HTML-страницу с встроенным содержимым документа → страница отправляется клиенту → на клиенте данные доступны через `window.__INITIAL_CONTENT__` для инициализации редактора.
[
   - **Требование:** Поток получения Wiki-страницы
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.5, US2.2.5]
]

6. **Редактирование и сохранение страницы** – пользователь редактирует документ в Tiptap Editor → при сохранении JavaScript отправляет `PUT /api/documents/{id}` с новым Markdown и заголовком `X-XSRF-TOKEN` → REST-контроллер принимает запрос → сервис сохраняет файл в Git-репозиторий (JGit создаёт коммит) → записывает метаданные коммита в PostgreSQL → если включено уведомление – асинхронно отправляет email через SMTP → возвращает JSON с результатом операции.
[
   - **Требование:** Процесс сохранения (Git commit + уведомления)
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US2.2.1, US2.2.5, US4.3.1, US1.1.4]
]

7. **Попытка удаления документа с дочерними** – пользователь нажимает «Удалить» на странице документа → форма отправляется POST на `/documents/{id}/delete` с CSRF-токеном → сервис проверяет наличие дочерних документов:
   - Для soft-delete: ищет дочерние в статусах `Draft` или `Published`.
   - Для hard-delete: ищет дочерние в любом статусе (включая `Deleted`).
   → Если дочерние найдены — сервис возвращает ошибку → контроллер добавляет сообщение в модель → Thymeleaf рендерит страницу с сообщением: «Невозможно удалить документ: у него есть дочерние документы. Сначала удалите или переместите их.»

8. **Создание нового документа** – пользователь переходит на `/documents/new` → MVC-контроллер возвращает форму создания с выбором пространства и типа документа → после заполнения форма отправляется через POST на `/documents` → сервис создаёт новый `.md`-файл в Git-репозитории → сохраняет метаданные в PostgreSQL → выполняет редирект на страницу созданного документа.
При установке родительского документа (`parent_document_id`) система проверяет, что `space_id` дочернего документа совпадает с `space_id` родительского. При несовпадении возвращается ошибка валидации.
[
   - **Требование:** Поток создания документа
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.1.2, US1.1.6, US2.1.1]
]

9. **Загрузка вложения** – пользователь добавляет изображение/файл через Tiptap Editor → JavaScript отправляет `POST /api/blobs` (multipart) с заголовком `X-XSRF-TOKEN` → сервис сохраняет файл в Blob-хранилище → записывает ссылку в PostgreSQL → возвращает URL для вставки в Markdown.
[
   - **Требование:** Процесс загрузки вложений
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US1.4.1, US1.4.2]
]

10. **Поиск** – пользователь вводит поисковый запрос в форму → форма отправляется методом GET на `/search` с параметром `q` → MVC-контроллер вызывает сервис поиска → сервис выполняет поиск по индексам в PostgreSQL → контроллер добавляет результаты в модель → Thymeleaf рендерит HTML-страницу с результатами поиска.
[
   - **Требование:** Поток поиска
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US3.1.1, US3.1.2, US3.1.3]
]

11. **Просмотр истории версий** – пользователь переходит на `/documents/{id}/history` → MVC-контроллер через JGit получает историю коммитов файла из Git-репозитория → получает метаданные коммитов из PostgreSQL → добавляет данные в модель → Thymeleaf рендерит HTML-страницу с историей изменений.
[
   - **Требование:** Поток просмотра истории
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US2.2.2]
]

12. **Сравнение версий** – пользователь выбирает две версии на странице истории → переходит на `/documents/{id}/diff?from={hash1}&to={hash2}` → MVC-контроллер через JGit получает diff между коммитами → добавляет результат в модель → Thymeleaf рендерит HTML-страницу с визуальным сравнением версий.
[
   - **Требование:** Поток сравнения версий
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US2.2.3, US2.2.6]
]

13. **Откат версии** – пользователь на странице истории нажимает "Восстановить версию" → форма отправляется POST на `/documents/{id}/restore/{commitHash}` с CSRF-токеном → сервис через JGit выполняет откат файла к указанному коммиту → создаёт новый коммит с информацией об откате → записывает метаданные в PostgreSQL → отправляет уведомление подписанным пользователям → выполняет редирект на страницу документа.
[
   - **Требование:** Поток отката версии
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US2.2.4, US2.2.5, US4.3.1]
]

14. **Управление пространствами** – администратор переходит на `/admin/spaces` → MVC-контроллер получает список пространств из PostgreSQL → Thymeleaf рендерит страницу управления → создание/редактирование пространств выполняется через HTML-формы с POST/PUT запросами → после сохранения выполняется редирект на список пространств. При настройке прав доступа администратор может назначать права как отдельным пользователям, так и группам пользователей.
[
   - **Требование:** Поток управления пространствами
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.2.1, US4.2.2]
]
[
   - **Требование:** Назначение прав группам на пространства
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.7]
   - **Комментарий:** При настройке прав администратор выбирает вкладку "Группы" и назначает права группе через space_group_permissions.
]

15. **Управление пользователями** – администратор переходит на `/admin/users` → MVC-контроллер получает список пользователей из PostgreSQL → Thymeleaf рендерит страницу управления → создание/редактирование пользователей и назначение ролей выполняется через HTML-формы → после сохранения выполняется редирект на список пользователей.
[
   - **Требование:** Поток управления пользователями
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.4, US4.1.6]
]

16. **Выход из системы (Logout)** – пользователь нажимает "Выйти" → форма отправляется POST на `/logout` с CSRF-токеном → сервер удаляет Cookie `JWT` и `XSRF-TOKEN` (устанавливает пустые значения с истёкшим сроком действия) → выполняет редирект на страницу логина.
[
   - **Требование:** Поток выхода из системы
   - **Статус:** Покрыто
   - **Элементы бэклога:** [US4.1.2]
]

17. **Управление группами пользователей** – администратор переходит на `/admin/groups` → MVC-контроллер получает список групп из PostgreSQL → Thymeleaf рендерит страницу управления с таблицей групп → создание/редактирование групп выполняется через HTML-формы → при удалении группы связи с пользователями и права группы на пространства удаляются каскадно, пользователи не затрагиваются → для управления составом администратор переходит на страницу состава группы → контроллер получает список участников группы и список всех пользователей → Thymeleaf рендерит интерфейс с возможностью добавления/исключения пользователей.
[
   - **Требование:** Поток управления группами пользователей
   - **Статус:** Покрыто
