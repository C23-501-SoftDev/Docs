# Логическая схема системы

В данном файле представлена логическая схема системы, описанной в [base-requirements.md](base-requirements.md)

## 1. Общая структура

### 1.1. Фронтенд (Frontend – React)

- Предоставляет пользовательский интерфейс: просмотр и редактирование Wiki-страниц, управление справочником пользователей, поиск.
- Реализован с использованием **React**.
- Включает встраиваемый **Tiptap Editor** – WYSIWYG-редактор с поддержкой Markdown, двухпанельным интерфейсом (исходный код/результат) и версионированием.
- Взаимодействует с сервером по протоколу HTTP с помощью клиента **Axios**, который сохраняет JWT-токен в `localStorage` и автоматически подставляет его в заголовок `Authorization`.
- Получает от сервера HTML-страницы, сформированные шаблонизатором, статические ресурсы (CSS, JS) и динамические данные (JSON).

### 1.2. Бэкенд (Backend – Spring Boot)

Реализован на **Java** с использованием **Spring Boot**, сборка – **Maven**.

#### 1.2.1. Spring MVC

- Формирует и отдаёт на фронтенд HTML-страницы (View) с помощью шаблонизатора.
- Обрабатывает HTTP-запросы, маршрутизацию.
- Отвечает за передачу статических ресурсов (CSS, JS).

#### 1.2.2. Spring Security + JWT

- Обеспечивает аутентификацию и авторизацию.
- При логине проверяет учётные данные (логин + хеш пароля) и выдает JWT.
- Для каждого запроса выполняет валидацию JWT-токена (встроенный Validator).

#### 1.2.3. REST API

- Контроллеры (`@RestController`) и DTO для обмена данными.
- Принимает AJAX-запросы от фронтенда, возвращает JSON или статусы HTTP.
- Взаимодействует с бизнес-логикой.

#### 1.2.4. Чистая архитектура (Clean Architecture)

Архитектура построена по принципу Clean Architecture (или аналогичному), с явным разделением на слои:

- **Domain (ядро):** Сущности (`Document`, `Space`, `User`), интерфейсы репозиториев (`DocumentRepository`, `SpaceRepository`). Не зависит от Spring и фреймворков.
- **Use Cases:** Сценарии использования (`CreateDocumentUseCase`, `UpdateDocumentUseCase`, `GetDocumentHistoryUseCase`). Содержат бизнес-логику, работают через интерфейсы репозиториев.
- **Interface (адаптеры):** REST Controllers, Presenters. Преобразуют запросы/ответы между внешним миром и Use Cases.
- **Infrastructure:** Реализации репозиториев (Spring Data JPA, JGit), отправка email, конфигурации.

#### 1.2.5. JGit (Git операции)

- Выполняет низкоуровневые Git-операции (коммиты, просмотр истории, откат, ветвление).
- Работает с Git-репозиторием, где хранятся `.md`-файлы Wiki.

#### 1.2.6. Async Email

- Асинхронная отправка уведомлений (например, при изменении страницы, регистрации).
- Взаимодействует с внешним SMTP-сервером (Mail.ru).

#### 1.2.7. Repository (Spring Data JPA)

- Слой доступа к реляционной базе данных.
- CRUD-операции над сущностями (пользователи, метаданные Git, теги, категории, настройки).

### 1.3. Хранилище (Storage)

#### 1.3.1. PostgreSQL 18

- Реляционная СУБД.
- Хранит метаданные о версиях (хэши коммитов, авторы, даты, сообщения), индексы для поиска, справочник пользователей, права доступа, теги, категории и вспомогательные данные.

#### 1.3.2. Git репозиторий

- Физическое хранилище Markdown-файлов (`.md`).
- Каждый документ – отдельный файл.
- Полное версионирование через Git (история, откат, сравнение версий).

#### 1.3.3. Blob хранилище

- Отдельная папка/директория для бинарных файлов: изображений, вложений (PDF, архивы и т.д.).
- Ссылки на blob-объекты хранятся в метаданных PostgreSQL.

### 1.4. Внешний SMTP-сервер

- Используется SMTP-сервер Mail.ru (или другой провайдер).
- Принимает команды отправки от асинхронного почтового сервиса бэкенда.

## 2. Потоки данных

1. **Инициализация** – пользователь заходит на сайт → сервер через Spring MVC формирует и отдаёт HTML-страницу (View) с React-приложением.
2. **Аутентификация** – пользователь вводит логин и пароль → фронтенд отправляет пароль на `/api/login` → Spring Security проверяет данные через БД → формирует JWT → возвращает токен клиенту → Axios сохраняет токен в `localStorage`.
3. **Защищённые запросы** – ко всем последующим API-вызовам Axios автоматически добавляет `Authorization: Bearer <JWT>` → сервер валидирует токен через Spring Security → при успехе выполняет бизнес-логику, иначе – 401/403.
4. **Получение Wiki-страницы** – клиент запрашивает `/api/pages/{id}` → контроллер вызывает `DocumentService` → сервис через JGit читает соответствующий `.md`-файл из Git-репозитория → возвращает Markdown → фронтенд отображает его в Tiptap (двухпанельно).
5. **Сохранение страницы** – пользователь редактирует в Tiptap → клиент отправляет `PUT /api/pages/{id}` с новым Markdown → сервис сохраняет файл в Git-репозиторий (JGit создаёт коммит) → записывает метаданные коммита в PostgreSQL → если включено уведомление – асинхронно отправляет email через SMTP.
6. **Загрузка вложения** – пользователь добавляет изображение/файл → клиент отправляет `POST /api/blobs` (multipart) → сервис сохраняет файл в Blob-хранилище → записывает ссылку в БД → возвращает URL для вставки в Markdown.
7. **Поиск** – пользователь вводит поисковый запрос → клиент вызывает `/api/search?q=...` → сервис выполняет полнотекстовый поиск по индексам в PostgreSQL → возвращает список результатов.
