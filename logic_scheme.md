# Логическая схема системы

В данном файле представлена логическая схема системы, описанной в [base-requirements.md](base-requirements.md)

## 1. Общая структура

### 1.1. Фронтенд (Client)

- Предоставляет пользовательский интерфейс: просмотр и редактирование Wiki-страниц, управление справочником пользователей, поиск. Клиентская часть реализована с использованием **Spring MVC**.
- Включает встраиваемый WYSIWYG-редактор для работы с Wiki-контентом.
- Взаимодействует с сервером по протоколу HTTP.
- Получает от сервера HTML-страницы, сформированные шаблонизатором, статические ресурсы (CSS, JS) и динамические данные (JSON).

### 1.2. Бэкенд

Состоит из следующих ключевых компонентов:

#### 1.2.1. Сервер (Server)

- Центральный узел обработки запросов, реализованный на **Java** с использованием фреймворка **Spring Boot**. Сборка проекта осуществляется с помощью **Maven**.
- Отвечает за маршрутизацию, бизнес-логику, интеграцию с остальными компонентами.
- Взаимодействует:

  - с фронтендом (формирует View и отдаёт статику, а также API-ответы),
  - с файловой системой (SMB-шара),
  - с базой данных,
  - с подсистемой аутентификации/токенов.

#### 1.2.2. Файловая система (File System) + Git

- Физическое хранилище Wiki-файлов (текстовые файлы в формате Markdown, изображения, вложения).
- Доступ организован по протоколу SMB.
- Поверх файлов настроен Git, обеспечивающий версионность каждой страницы (возможность просмотра истории изменений, отката к предыдущим версиям, ветвления).

#### 1.2.3. База данных (Database)

- В качестве реляционной СУБД используется **PostgreSQL 18**.
- Хранит служебную информацию:

  - метаданные о Git-версиях (хэши коммитов, авторы, даты, комментарии),
  - индексы для быстрого поиска,
  - структуры данных (справочник пользователей, права доступа, теги, категории),
  - настройки и вспомогательные данные.
- Взаимодействует с сервером для выборки и обновления данных.

### 1.3. Подсистема аутентификации и токенов (Auth/Token Service)

- Отдельный компонент, расположенный вне бэкенда, отвечающий за генерацию и валидацию токенов доступа (JWT).
- Состоит из двух логических частей:

  - **Issuer (выдача)** – генерирует токены после успешной аутентификации пользователя. Расположен на стороне сервера (API).
  - **Validator (валидация)** – проверяет подлинность токенов при каждом запросе к защищённым ресурсам. Может быть реализован как middleware на сервере или как отдельный сервис.
- Обеспечивает безопасный доступ к API и разграничение прав пользователей на основе ролей и политик.

## 2. Потоки данных

1. Пользователь заходит на сайт → сервер через Spring MVC формирует и отдает HTML-страницу (View).
2. При попытке входа фронтенд отправляет учётные данные на сервер → сервер проверяет их через базу данных и обращается к подсистеме выдачи токенов (Issuer), которая возвращает JWT-токен. Токен передаётся клиенту.
3. Все последующие запросы к API фронтенд выполняет с токеном в заголовке Authorization. Сервер передаёт токен подсистеме валидации (Validator); при успешной проверке запрос обрабатывается, иначе возвращается ошибка 401/403.
4. При работе с Wiki-страницами:
   - Запрос на получение содержимого → сервер извлекает файл из файловой системы (SMB) и отдаёт его клиенту. Фронтенд отображает содержимое во встроенном WYSIWYG-редакторе.
   - Сохранение изменений → фронтенд отправляет обновлённое содержимое на сервер через API. Сервер сохраняет файл в файловую систему, фиксирует изменения через Git (создаёт коммит) и записывает метаданные коммита в базу данных.
5. Поиск выполняется по индексам, хранящимся в базе данных; сервер возвращает результаты.
